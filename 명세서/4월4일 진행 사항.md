Layer 0 — Step 1: Engine API 확장 (Getter 추가)

왜 필요한가 (근거)

Editor의 Hierarchy 패널은 "현재 레벨에 어떤 레이어가 있고, 각 레이어에 어떤 오브젝트가 있는지" 목록을 표시해야 해. 그런데 현재 Engine 구조를 보면:

- CObject_Manager::m_pLayers → private, getter 없음
- CLayer::m_GameObjects → private list<CGameObject\*>, getter 없음
- CLevel_Manager::m_iCurrentLevelIndex → private, getter 없음
- CGameObject::m_Components → protected, 외부 접근 불가

즉, Editor가 아무리 CGameInstance를 통해 Engine에 접근하려 해도 읽을 수 있는 인터페이스 자체가 없는 상태야. "문은 있는데 손잡이가 없는" 상황.

왜 이 방식인가 (이유)

대안 1: friend class로 Editor에게 직접 접근 허용

- 단점: Engine이 Editor를 알게 됨 → DLL 의존성 역전. Engine은 하위 프로젝트를 몰라야 하는 원칙 위반

대안 2: public 멤버로 변경

- 단점: 캡슐화 완전 파괴. 아무 곳에서나 수정 가능해져 버그 추적이 어려워짐

채택: const reference/pointer 반환 getter

- const 반환이므로 읽기만 가능, 수정 불가 → 캡슐화 유지
- Engine 기존 코드 변경 최소화 (getter 한 줄 추가)
- CGameInstance 래퍼를 통해 노출하므로 DLL export 경계도 깔끔

개념: DLL Export 경계와 래퍼 패턴

우리 프로젝트는 Engine이 DLL이야. Editor가 Engine 내부 클래스에 접근하려면 반드시 ENGINE_DLL이 붙은 클래스를 통해야 해.

Editor.exe → CGameInstance (ENGINE_DLL) → CObject_Manager (내부) → CLayer (내부)

CObject_Manager나 CLayer는 ENGINE_DLL export가 아니야. 그래서 Editor에서 직접 CObject_Manager::Get_Layers()를 호출할 수 없어. 반드시 CGameInstance가
중간에서 래퍼(wrapper) 메서드로 전달해줘야 해. 이건 기존 패턴과 동일해 — 지금도 Add_GameObject, Change_Level 등이 전부 GameInstance 래퍼를 통해 호출되고
있어.

구체적 추가 내용

1-1. Layer.h — 오브젝트 리스트 읽기

// Layer.h — public 영역에 추가
const list<class CGameObject\*>& Get_GameObjects() const { return m_GameObjects; }

왜 const list& 반환인가: 복사 비용 없이(참조) 읽기 전용(const)으로 제공. Hierarchy는 매 프레임 이 리스트를 순회하며 이름을 표시하기 때문에, 매번 복사하면
성능 낭비.

1-2. Object_Manager.h/.cpp — 특정 레벨의 레이어 맵 접근

// Object_Manager.h — public 영역에 추가
const map<const \_wstring, class CLayer*>* Get_Layers(\_uint iLevelIndex) const;

// Object_Manager.cpp — 구현
const map<const \_wstring, class CLayer*>* CObject_Manager::Get_Layers(\_uint iLevelIndex) const
{
if (iLevelIndex >= m_iNumLevels)
return nullptr;

      return &m_pLayers[iLevelIndex];

}

왜 포인터 반환인가: m_pLayers는 동적 배열(new map[iNumLevels])이야. 잘못된 인덱스가 들어오면 nullptr을 반환해서 호출부에서 null 체크할 수 있게 하기 위함.
reference는 "없음"을 표현할 수 없으니까.

1-3. Level_Manager.h — 현재 레벨 인덱스

// Level_Manager.h — public 영역에 추가
\_int Get_CurrentLevelIndex() const { return m_iCurrentLevelIndex; }

왜 필요한가: Get_Layers(iLevelIndex)를 호출하려면 "지금 몇 번째 레벨인지" 알아야 해. 현재 이 값은 CLevel_Manager 안에 -1로 초기화되어 있고, Change_Level 시
갱신돼. Editor가 이걸 읽어서 Get_Layers(현재레벨인덱스) 형태로 호출하게 됨.

1-4. GameObject.h — 컴포넌트 맵, Transform 접근

// GameObject.h — public 영역에 추가
const map<const \_wstring, CComponent*>& Get_Components() const { return m_Components; }
CTransform* Get_Transform() const { return m_pTransformCom; }

왜 두 개인가:

- Get_Components(): Inspector에서 "이 오브젝트가 가진 모든 컴포넌트"를 순회하며 RTTR로 프로퍼티를 표시할 때 사용
- Get_Transform(): Transform은 모든 오브젝트가 반드시 갖는 컴포넌트라서 자주 접근됨. 매번 Get_Components()에서 찾는 대신 직접 접근하는 편의 메서드.
  ImGuizmo 조작, Hierarchy 선택 시 위치 표시 등에서 반복 사용될 예정

1-5. GameInstance.h/.cpp — 래퍼 메서드

// GameInstance.h — #pragma region OBJECT_MANAGER 안에 추가
const map<const \_wstring, class CLayer*>* Get_Layers(\_uint iLevelIndex) const;

// GameInstance.h — #pragma region LEVEL_MANAGER 안에 추가
\_int Get_CurrentLevelIndex() const;

// GameInstance.cpp
const map<const \_wstring, class CLayer*>* CGameInstance::Get_Layers(\_uint iLevelIndex) const
{
return m_pObject_Manager->Get_Layers(iLevelIndex);
}

\_int CGameInstance::Get_CurrentLevelIndex() const
{
return m_pLevel_Manager->Get_CurrentLevelIndex();
}

호출 흐름 예시 (Hierarchy에서 사용할 때):

CPanel_Hierarchy::Render()
→ CGameInstance::Get_CurrentLevelIndex() // 현재 레벨 번호
→ CGameInstance::Get_Layers(iLevelIndex) // 해당 레벨의 레이어 맵
→ 각 CLayer에서 Get_GameObjects() // 각 레이어의 오브젝트 리스트
→ 각 CGameObject에서 Get_Name() // 표시할 이름

Layer 0 — Step 2: 오브젝트 이름/태그 시스템

왜 필요한가 (근거)

현재 CGameObject에는 자기 자신을 식별할 수 있는 문자열이 없어. Hierarchy에서 오브젝트 목록을 보여주려면 "이 오브젝트가 뭔지" 표시할 이름이 필요해. 이름
없이는 목록에 "GameObject 0x00FF..." 같은 주소밖에 보여줄 게 없어.

또한 **태그(Tag)**는 같은 종류의 오브젝트를 그룹으로 묶어 검색/필터링할 때 쓰여. 예: 태그가 "Monster"인 오브젝트만 필터링, "NPC"만 선택 등. Unity의 Tag
시스템과 같은 개념이야.

왜 이 방식인가 (이유)

대안 1: Prototype 태그를 그대로 이름으로 사용

- 문제: 같은 프로토타입에서 Clone된 오브젝트들이 전부 같은 이름이 됨. "ForkLift"가 5개면 5개 다 "ForkLift" → 구분 불가

대안 2: 자동 넘버링 ("ForkLift_001", "ForkLift_002")

- 가능하지만 이름 변경 기능이 없으면 의미 없음. 우선은 기본 이름 + 수동 Set_Name으로 충분

채택: 멤버 변수 + getter/setter

- m_strName: 개별 인스턴스의 고유 표시 이름. 기본값은 프로토타입 태그 복사
- m_strTag: 분류/검색용 태그. 선택적 사용
- 단순한 구조지만, Hierarchy/Inspector/씬 저장 시 모두 이 이름을 기준으로 참조하게 되므로 초기부터 확립해야 나중에 리팩토링 안 해도 됨

개념: Name vs Tag의 역할 구분

┌────────┬────────────────────────────────┬────────────────────────────────┐
│ │ Name │ Tag │
├────────┼────────────────────────────────┼────────────────────────────────┤
│ 용도 │ UI 표시, 개별 식별 │ 그룹 분류, 검색/필터 │
├────────┼────────────────────────────────┼────────────────────────────────┤
│ 고유성 │ 고유하면 좋지만 강제는 않음 │ 여러 오브젝트가 같은 태그 가능 │
├────────┼────────────────────────────────┼────────────────────────────────┤
│ 예시 │ "Boss_JinWoo", "SpawnPoint_03" │ "Monster", "NPC", "Light" │
├────────┼────────────────────────────────┼────────────────────────────────┤
│ 참고 │ Unity의 gameObject.name │ Unity의 gameObject.tag │
└────────┴────────────────────────────────┴────────────────────────────────┘

구체적 추가 내용

// GameObject.h — protected 멤버 추가 (m_iFlag 아래)
\_wstring m_strName = TEXT("");
\_wstring m_strTag = TEXT("");

// public 접근자
public:
const \_wstring& Get_Name() const { return m_strName; }
void Set_Name(const \_wstring& strName) { m_strName = strName; }
const \_wstring& Get_Tag() const { return m_strTag; }
void Set_Tag(const \_wstring& strTag) { m_strTag = strTag; }

이름 초기 설정: CObject_Manager::Add_GameObject에서 Clone 직후, 프로토타입 태그를 기본 이름으로 복사하면 됨. 나중에 Editor Inspector에서 이름을 수동 변경할
수 있게 되면 Set_Name()으로 덮어쓰기.

Layer 0 — Step 3: RTTR 라이브러리 세팅

왜 필요한가 (근거)

Inspector 패널은 선택된 오브젝트의 컴포넌트 프로퍼티를 자동으로 UI 위젯으로 표시해야 해. 예를 들어 CTransform_3D를 선택하면
Position(float3), Rotation(float3), Scale(float3)이 자동으로 DragFloat3 위젯으로 나와야 해.

이걸 하드코딩으로 하면:
// 하드코딩 방식 — 컴포넌트 추가할 때마다 Inspector 코드 수정 필요
if (컴포넌트가 Transform_3D면)
DragFloat3("Position", ...);
DragFloat3("Rotation", ...);
else if (컴포넌트가 Texture면)
...
컴포넌트가 늘어날 때마다 Inspector 코드에 분기를 계속 추가해야 하고, 프로퍼티 하나 바뀌면 Inspector도 같이 수정해야 해.
확장성이 매우 낮아.

**RTTR(Run Time Type Reflection)**을 쓰면:

- 컴포넌트의 프로퍼티를 한 곳(RTTR_Registration.cpp)에서 등록만 하면
- Inspector가 등록된 프로퍼티를 자동 순회해서 타입에 맞는 위젯을 생성
- 새 컴포넌트 추가 시 Inspector 코드는 수정할 필요 없음, 등록만 추가

RTTR이란? (개념)

**Reflection(리플렉션)**이란 프로그램이 런타임에 자기 자신의 타입 정보를 조회할 수 있는 기능이야. Java나 C#은 언어 차원에서
지원하지만, C++은 기본 지원이 없어. RTTR은 C++에 이 기능을 제공하는 오픈소스 라이브러리야.

핵심 기능:

- 타입 등록: registration::class\_<CTransform_3D>("CTransform_3D") → "이런 클래스가 있다"
- 프로퍼티 등록: .property("Position", &CTransform_3D::Get_State, &CTransform_3D::Set_State) → "이 클래스에 이런 프로퍼티가
  있다"
- 런타임 조회: type::get<CTransform_3D>().get_properties() → 등록된 프로퍼티 목록 순회
- 타입→위젯 매핑: float3 → ImGui::DragFloat3, bool → ImGui::Checkbox 등 자동 대응

왜 "외부 등록" 방식인가 (이유)

RTTR 등록 코드를 어디에 둘지 두 가지 선택지가 있어:

방식 A: 클래스 내부에 등록 (침투적)
// Engine/Private/Transform_3D.cpp 에 RTTR 매크로 추가
RTTR_REGISTRATION { ... }

- 문제: Engine에 RTTR 의존성이 생김. Engine은 RTTR을 모르게 하기로 결정했으므로 위반

방식 B: Editor에서 외부 등록 (비침투적) ← 채택
// Editor/Private/RTTR_Registration.cpp 에서 Engine 클래스를 외부에서 등록
#include <rttr/registration>
#include "Transform_3D.h"

RTTR*REGISTRATION {
registration::class*<CTransform_3D>("CTransform_3D")
.property("Position", ...)
...;
}

- Engine/Client 코드에 RTTR include/매크로 일절 없음
- RTTR 헤더, lib, dll 모두 Editor 프로젝트에만 추가
- 등록할 프로퍼티를 변경할 때 Editor만 수정

세팅 절차

3-1. RTTR 라이브러리 획득

RTTR 공식 사이트(rttr.org)에서 빌드된 바이너리 또는 소스를 받아. 필요한 파일:

Editor/
├── RTTR/
│ ├── include/ ← rttr/registration.h, rttr/type.h 등 헤더
│ └── lib/
│ ├── Debug/ ← rttr_core_d.lib, rttr_core_d.dll
│ └── Release/ ← rttr_core.lib, rttr_core.dll

3-2. Editor 프로젝트 설정 (Visual Studio)

┌──────────────────────────┬──────────────────────────────────────────────────────────┐
│ 항목 │ 설정 │
├──────────────────────────┼──────────────────────────────────────────────────────────┤
│ 추가 포함 디렉터리 │ $(SolutionDir)Editor/RTTR/include                        │
  ├──────────────────────────┼──────────────────────────────────────────────────────────┤
  │ 추가 라이브러리 디렉터리 │ $(SolutionDir)Editor/RTTR/lib/$(Configuration) │
├──────────────────────────┼──────────────────────────────────────────────────────────┤
│ 추가 종속성 │ rttr_core_d.lib (Debug) / rttr_core.lib (Release) │
├──────────────────────────┼──────────────────────────────────────────────────────────┤
│ DLL 복사 │ 빌드 후 이벤트로 rttr_core_d.dll → 실행 파일 경로에 복사 │
└──────────────────────────┴──────────────────────────────────────────────────────────┘

3-3. CRT 주의

RTTR을 빌드할 때 CRT 설정이 우리 프로젝트와 일치해야 해. 우리 프로젝트는 /MDd(Debug), /MD(Release) 동적 CRT를 사용 중이야.
RTTR도 동일한 CRT로 빌드된 바이너리를 사용하거나, 소스에서 같은 설정으로 빌드해야 해. 불일치 시 힙 충돌(heap corruption)이
발생해 — 이건 CLAUDE.md에도 "CRT 통일 필수"로 명시된 기존 원칙이야.

3-4. 확인 테스트

세팅 완료 후 Editor 코드에서 간단한 테스트:

#include <rttr/registration.h>
#include <rttr/type.h>

// 어딘가에서 (예: EditorApp::Initialize)
rttr::type t = rttr::type::get_by_name("CTransform_3D");
// → RTTR_Registration.cpp에 등록 전이면 invalid type, 등록 후면 유효

이게 빌드+실행되면 RTTR 세팅 완료.

Step 4: RTTR_Registration.cpp — 외부 등록 파일

근거(Why)

Inspector 패널이 오브젝트의 프로퍼티를 자동으로 UI 위젯으로 표시하려면, 런타임에 "이 클래스에 어떤 멤버가 있는가?"를 알아야
합니다. C++은 Java/C#과 달리 네이티브 리플렉션이 없으므로, RTTR 라이브러리를 통해 수동으로 타입 정보를 등록해야 합니다.

이유(Rationale) — 왜 "외부 등록" 방식인가?

RTTR 등록 방식은 크게 두 가지입니다:

┌──────────┬────────────────────────────────────────────────────┬──────────────────────────────────────────────────────┐
│ 방식 │ 코드 위치 │ 장단점 │
├──────────┼────────────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
│ 내부 │ Engine 클래스 .cpp 파일 안에 RTTR_REGISTRATION { │ Engine이 RTTR에 의존하게 됨. Client/GameApp도 RTTR │
│ 등록 │ ... } │ 링크 필요 │
├──────────┼────────────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
│ 외부 │ Editor/Private/RTTR_Registration.cpp │ Engine/Client 코드 수정 없음. RTTR 의존성이 │
│ 등록 │ │ Editor에만 격리 │
└──────────┴────────────────────────────────────────────────────┴──────────────────────────────────────────────────────┘

우리는 외부 등록을 선택합니다. 이유:

- Engine/Client DLL에 RTTR 의존성을 추가하면 GameApp까지 RTTR DLL이 필요해짐
- Engine 클래스의 public getter/setter만 사용하므로 friend 선언이 불필요
- 새 컴포넌트 추가 시 이 파일에 등록만 추가하면 Inspector가 자동 반영

개념(Concept) — RTTR 외부 등록

RTTR*REGISTRATION
{
rttr::registration::class*<CTransform_3D>("CTransform_3D")
.property("Position", &CTransform_3D::Get_State, &CTransform_3D::Set_State)
...
}

RTTR의 RTTR_REGISTRATION 매크로는 전역 정적 초기화(static initialization) 를 이용합니다. 프로그램 시작 시 main() 호출 전에
자동 실행되어 타입 정보가 RTTR 내부 레지스트리에 등록됩니다.

하지만 문제가 있습니다: CTransform의 getter/setter는 XMVECTOR, \_fvector 같은 DirectXMath 타입을 사용합니다. RTTR은 이런
SIMD 타입을 직접 다룰 수 없고, Inspector UI에도 적합하지 않습니다.

해결 방안: Wrapper 접근자 + \_float3/\_float4x4

RTTR에는 plain 타입(\_float3, \_float, \_bool 등)을 등록하고, RTTR 등록 시 래퍼 람다/자유 함수로 XMVECTOR ↔ \_float3 변환을
처리합니다.

Inspector가 보여줄 프로퍼티:
Position → \_float3 (DragFloat3)
Scale → \_float3 (DragFloat3)
Rotation → \_float3 (오일러 각도, DragFloat3)
Speed → \_float (DragFloat)
RotSpeed → \_float (DragFloat)

구현 코드

Editor/Private/RTTR_Registration.cpp:

#include "Editor_Defines.h"

// RTTR
#include <rttr/registration.h>

// Engine headers
#include "Transform.h"
#include "Transform_3D.h"
#include "Transform_2D.h"
#include "GameObject.h"
#include "Camera.h"

using namespace Engine;

// ============================================================
// Helper: Transform 프로퍼티 접근을 위한 래퍼 함수
// ============================================================
// Why: CTransform의 Get_State()는 XMVECTOR를 반환하고
// Set_State()는 \_fvector를 받는다.
// RTTR은 SIMD 타입을 직접 처리할 수 없으므로
// \_float3 ↔ XMVECTOR 변환 래퍼가 필요하다.
// ============================================================

namespace RTTR_Helpers
{
// --- Position ---
\_float3 GetPosition(CTransform* pTransform)
{
\_float3 vPos;
XMStoreFloat3(&vPos, pTransform->Get_State(STATE::POSITION));
return vPos;
}
void SetPosition(CTransform* pTransform, const \_float3& vPos)
{
pTransform->Set_State(STATE::POSITION, XMVectorSetW(XMLoadFloat3(&vPos), 1.f));
}

      // --- Scale ---
      _float3 GetScale(CTransform* pTransform)
      {
          return pTransform->Get_Scale();
      }
      void SetScale(CTransform* pTransform, const _float3& vScale)
      {
          pTransform->Set_Scale(vScale.x, vScale.y, vScale.z);
      }

      // --- Rotation (오일러 각도, Degree) ---
      // Why: WorldMatrix에서 오일러 각도를 추출하는 것은
      //      Gimbal Lock 등의 문제가 있지만,
      //      Inspector에서 직관적으로 회전을 표시/수정하려면
      //      오일러 각도가 가장 적합하다.
      //      정밀한 회전은 Viewport의 Gizmo로 처리한다.
      _float3 GetRotation(CTransform* pTransform)
      {
          _float3 vScale = pTransform->Get_Scale();
          _float4x4 world = *pTransform->Get_WorldMatrixPtr();

          // 스케일 제거 → 순수 회전 행렬 추출
          _float3 r0 = { world._11 / vScale.x, world._12 / vScale.x, world._13 / vScale.x };
          _float3 r1 = { world._21 / vScale.y, world._22 / vScale.y, world._23 / vScale.y };
          _float3 r2 = { world._31 / vScale.z, world._32 / vScale.z, world._33 / vScale.z };

          _float pitch, yaw, roll;

          // XZY 순서 분해 (DirectX 좌표계 기준)
          pitch = asinf(-r2.y);  // X축 회전

          if (fabsf(cosf(pitch)) > 0.001f)
          {
              yaw  = atan2f(r2.x, r2.z);   // Y축 회전
              roll = atan2f(r0.y, r1.y);    // Z축 회전
          }
          else
          {
              yaw  = atan2f(-r0.z, r0.x);
              roll = 0.f;
          }

          // Radian → Degree
          return _float3{
              XMConvertToDegrees(pitch),
              XMConvertToDegrees(yaw),
              XMConvertToDegrees(roll)
          };
      }
      void SetRotation(CTransform* pTransform, const _float3& vDegrees)
      {
          _float3 vScale = pTransform->Get_Scale();
          _float rad_x = XMConvertToRadians(vDegrees.x);
          _float rad_y = XMConvertToRadians(vDegrees.y);
          _float rad_z = XMConvertToRadians(vDegrees.z);

          // 오일러 각도 → 회전 행렬 → 스케일 적용 → WorldMatrix에 반영
          XMMATRIX matRot = XMMatrixRotationRollPitchYaw(rad_x, rad_y, rad_z);

          _float4x4 world = *pTransform->Get_WorldMatrixPtr();

          // Right, Up, Look 벡터에 스케일 * 회전 반영
          XMVECTOR vRight = XMVector3Normalize(matRot.r[0]) * vScale.x;
          XMVECTOR vUp    = XMVector3Normalize(matRot.r[1]) * vScale.y;
          XMVECTOR vLook  = XMVector3Normalize(matRot.r[2]) * vScale.z;

          pTransform->Set_State(STATE::RIGHT, vRight);
          pTransform->Set_State(STATE::UP,    vUp);
          pTransform->Set_State(STATE::LOOK,  vLook);
      }

}

// ============================================================
// RTTR 등록
// ============================================================
// Concept: RTTR_REGISTRATION 매크로는 전역 정적 변수를 생성하여
// main() 전에 자동 실행된다.
// 이 파일이 Editor.exe에 링크되는 순간 등록이 완료된다.
//
// Why 외부 등록: Engine/Client 코드에 RTTR 의존성 없이
// Editor에서만 타입 정보를 등록/사용한다.
// ============================================================

RTTR*REGISTRATION
{
// --- \_float3 (값 타입으로 등록) ---
// Why: Inspector에서 Position/Scale/Rotation을 DragFloat3로 표시하려면
// RTTR이 \_float3의 멤버(x, y, z)를 알아야 한다.
rttr::registration::class*<\_float3>("\_float3")
.property("x", &\_float3::x)
.property("y", &\_float3::y)
.property("z", &\_float3::z)
;

      // --- CTransform (기본 클래스) ---
      rttr::registration::class_<CTransform>("CTransform")
          .property("Position",
              &RTTR_Helpers::GetPosition,
              &RTTR_Helpers::SetPosition)
          .property("Scale",
              &RTTR_Helpers::GetScale,
              &RTTR_Helpers::SetScale)
          .property("Rotation",
              &RTTR_Helpers::GetRotation,
              &RTTR_Helpers::SetRotation)
          ;

      // --- CTransform_3D ---
      // Concept: base_class<CTransform>()로 상속 관계를 등록하면
      //          Inspector에서 CTransform_3D 인스턴스를 조회할 때
      //          부모 CTransform의 Position/Scale/Rotation 프로퍼티도
      //          자동으로 포함된다.
      rttr::registration::class_<CTransform_3D>("CTransform_3D")
          .constructor<>()(rttr::policy::ctor::as_raw_ptr)  // 메타데이터용, 실제 생성은 Prototype 패턴
          ;

      // --- CTransform_2D ---
      rttr::registration::class_<CTransform_2D>("CTransform_2D")
          .constructor<>()(rttr::policy::ctor::as_raw_ptr)
          ;

      // --- CGameObject ---
      rttr::registration::class_<CGameObject>("CGameObject")

}

주요 설계 포인트

1. 래퍼 함수가 필요한 이유

- CTransform::Get_State(STATE::POSITION)은 XMVECTOR를 반환 → SIMD 레지스터 타입이라 RTTR이 직접 저장/전달 불가
- \_float3 (= XMFLOAT3)는 일반 구조체이므로 RTTR이 처리 가능
- Inspector UI도 float[3] → ImGui::DragFloat3로 자연스럽게 매핑

2. Rotation 분해의 한계

- 월드 행렬에서 오일러 각도를 역추출하면 Gimbal Lock 문제 발생 가능
- 하지만 Inspector는 "대략적인 확인/미세 조정"용이고, 정밀 회전은 Viewport의 ImGuizmo로 처리할 예정
- Unity/Unreal도 Inspector에 오일러 각도를 표시하지만, 내부적으로는 쿼터니언을 사용

3. base_class<>() 미사용 이유

- RTTR의 base_class<CTransform>()는 기본 클래스의 프로퍼티를 자동 상속시키는 기능
- 그런데 CTransform_3D/2D의 생성자가 private이고 ID3D11Device\* 등을 받으므로, RTTR의 기본 생성자 등록이 까다로움
- 대신 Inspector에서 rttr::type::get_by_name("CTransform")으로 부모 프로퍼티를 직접 조회하는 방식이 더 안전

4. 향후 확장

- 새 컴포넌트(예: CModel, CCollider)를 추가하면 이 파일에 등록 블록만 추가
- Inspector 코드는 등록된 프로퍼티를 순회하므로 자동으로 새 UI가 생성됨

한 가지 확인이 필요합니다 — CTransform_3D/2D의 생성자가 private인데 constructor<>() 등록이 컴파일 에러를 낼 수 있습니다.
실제로 Inspector에서 RTTR로 객체를 생성하지는 않고 (이미 Prototype/Clone으로 생성된 객체의 프로퍼티만 읽기/쓰기), 생성자
등록은 제거해도 됩니다. 빌드 테스트 후 에러가 나면 .constructor<>() 줄을 삭제하세요.

❯ 왜 RTTR에서는 SIMD 타입 처리가 안되지?

● XMVECTOR는 실제로 \_\_m128입니다 — CPU의 SSE 레지스터를 직접 매핑하는 컴파일러 내장 타입(intrinsic) 이죠.

RTTR이 처리 못하는 이유는 두 가지입니다:

1. 정렬(Alignment) 문제

- \_\_m128은 16바이트 정렬이 필수
- RTTR은 프로퍼티 값을 rttr::variant에 담아서 전달하는데, variant 내부 저장소가 16바이트 정렬을 보장하지 않음
- 정렬이 깨진 상태에서 SSE 명령어가 접근하면 → 크래시

2. 일반 C++ 타입이 아님

- \_\_m128은 클래스/구조체가 아니라 하드웨어 레지스터 별칭
- 생성자, 복사 연산자 같은 C++ 의미론(semantics)이 없음
- RTTR의 variant는 값을 복사/이동/저장하면서 일반 C++ 타입처럼 다루는데, 레지스터 타입에는 이게 안전하지 않음

반면 \_float3 (XMFLOAT3)는:
struct XMFLOAT3 {
float x, y, z;
};
그냥 평범한 구조체입니다. 4바이트 정렬이면 충분하고, 복사도 자유롭고, RTTR의 variant에 안전하게 담깁니다.

그래서 래퍼 함수가 하는 일은 결국:
Inspector 읽기: XMVECTOR(레지스터) → XMStoreFloat3 → \_float3(메모리) → RTTR variant
Inspector 쓰기: RTTR variant → \_float3(메모리) → XMLoadFloat3 → XMVECTOR(레지스터)

DirectXMath가 XMStoreFloat3/XMLoadFloat3 같은 함수를 제공하는 것도 같은 이유 — 레지스터 ↔ 메모리 간 변환이 항상 필요하기
때문입니다.

## Troubleshooting 기록

### 1. `/permissive-` (표준 준수 모드) 문제

**증상**: bind_impl.h에서 C3878 `예기치 않은 토큰 '>'` 에러 대량 발생. include 순서나 `#undef new`와 무관하게 지속.

**원인**: VS 2022는 기본적으로 `/permissive-` 활성화. RTTR 0.9.6(2018)의 템플릿 메타프로그래밍 코드가 이 엄격 모드에서 호환 안 됨.

**해결**: Editor 프로젝트 속성 → C/C++ → 언어 → 표준 준수 모드 → **아니요** (`/permissive`)

**교훈**: `#define new DBG_NEW` 매크로 충돌이 아니었음. bind_impl.h에 `new` 키워드가 없다는 것을 grep으로 확인한 후 방향 전환.

### 2. RTTR 자유 함수 래퍼 등록 실패

**증상**: static_assert 3개 — "getter-member-function without arguments", "setter with one argument", "same signature"

**원인**: RTTR의 `.property(getter, setter)`는 **멤버 함수 포인터**를 기대. 자유 함수 `GetPosition(CTransform*)` 또는 `GetPosition(const CTransform&)`는 인자가 있으므로 RTTR이 멤버 함수로 인식 못함.

**해결**: CTransform에 `_float3` 기반 멤버 함수(Get_Position, Set_Position 등) 추가 → `&CTransform::Get_Position` 형태로 등록

### 3. Set_Scale 오버로드 모호성

**증상**: `&CTransform::Set_Scale` 에서 컴파일 에러 — 기존 `Set_Scale(_float, _float, _float)`과 새 `Set_Scale(_float3)` 중 어느 것인지 모호

**해결 방법 2가지**:
- `static_cast<void(CTransform::*)(_float3)>(&CTransform::Set_Scale)` — 시그니처 명시 캐스팅
- 함수 이름 분리 `Set_ScaleF3(_float3)` — 최종 채택 (단순함)

### 4. CTransform_3D/2D 등록 시 private 소멸자 접근 불가

**증상**: C2248 `~CTransform_3D()`: private 멤버에 액세스할 수 없습니다 (destructor_wrapper.h:57)

**원인**: RTTR이 `class_<T>` 등록 시 내부적으로 소멸자 래퍼를 생성. concrete 클래스의 소멸자가 private/protected이면 접근 불가.

**해결**: CTransform_3D/2D 등록 제거. abstract 클래스(CTransform)는 소멸자 래퍼를 생성하지 않아 문제없음. Inspector는 `CTransform*` 포인터로 접근하므로 부모 프로퍼티로 충분.

### 5. `#define new DBG_NEW` 와 RTTR 헤더 충돌 (초기 오해)

**증상**: bind_impl.h에서 동일한 C3878 에러

**초기 진단**: Engine_Defines.h의 `#define new DBG_NEW`가 RTTR의 placement new를 깨뜨린다고 판단

**실제 원인**: bind_impl.h에 `new` 키워드가 없음 → `#define new`는 무관. 진짜 원인은 `/permissive-` 모드

**교훈**: 에러 메시지만 보고 원인을 추정하지 말고, 실제 에러 발생 파일의 코드를 grep으로 확인할 것
