# 4월 20일 진행 사항

## 1. 현재 작업 맥락

Root Motion 1차 적용 후 `CAnimController` 기반의 애니메이션 제어 구조를 설계 중이다.

현재 방향은 다음과 같다.

- `CModel`은 모델/애니메이션 데이터와 Root Motion 추출을 담당한다.
- `CAnimController`는 Engine 범용 컴포넌트로 두고, 특정 게임 캐릭터/몬스터 개념을 알지 않는다.
- Client는 `Character`, `Monster` 단위로 애니메이션 액션 enum, 상태 정책, 애니메이션 테이블을 관리한다.
- `Loop`, `Root Motion` 여부는 `.bin` 파일에 저장된 애니메이션 데이터가 기준이다.
- Client 테이블은 `Action + 상태 축 -> AnimationName` 매핑만 담당한다.

---

## 2. Root Motion 관련 정리

현재 `SungJinWooERank.bin` 기준 Root Bone은 `Translate`로 설정하는 방향이 맞다.

이유:

- `Bip001`은 일부 자세 흔들림이나 Body Offset 성격의 이동이 섞여 있다.
- Dash, BackDash, 공격/스킬 계열의 실제 이동량은 대부분 `Translate`에 들어 있다.
- `Walk`, `Run`은 순이동이 없는 In-Place Cycle이다. 따라서 Transform 이동은 gameplay 이동 로직이 담당해야 한다.

주요 Root Motion 애니메이션:

| Animation | 성격 | Root | 이동 방향 |
|---|---|---|---|
| Dash | Root Motion | Translate | 전방 약 -6.330 |
| BackDash | Root Motion | Translate | 후방 약 +5.000 |
| BaseAttack_01 | Small Root Motion | Translate | 전방 약 -0.482 |
| BaseAttack_02 | Root Motion | Translate | 전방 약 -1.125 |
| BaseAttack_03 | Root Motion | Translate | 전방 약 -1.249 |
| Skill_01 | Root Motion | Translate | 전방 약 -4.427 |
| Skill_02 | Root Motion | Translate | 전방 약 -5.051 |
| Skill_01_Fighter | Small Root Motion | Translate | 전방 약 -0.890 |
| Skill_02_Fighter | Root Motion | Translate | 전방 약 -2.928 |
| GS_ChainSmash_01_OneHandSword | Root Motion | Translate | 전방 약 -8.558 |
| Breakfall | Root Motion | Translate | 후방 약 +5.401 |

주의:

- Non-loop 애니메이션은 끝났을 때 마지막 키프레임 pose를 유지해야 한다.
- 애니메이션 전환 시 현재/다음 애니메이션의 track position과 Root Motion baseline을 명시적으로 리셋해야 한다.
- 동일 one-shot 애니메이션을 다시 재생하려면 `Restart_Animation()` 또는 `bRestartOnEnter` 정책이 필요하다.
- 매 프레임 같은 one-shot 액션을 `Play()`하면 계속 첫 프레임으로 리셋될 수 있으므로, one-shot 액션은 상태 전환 순간에만 요청해야 한다.

## 3. CAnimController 설계 결정

`CAnimController`는 Engine에 두는 범용 `CComponent`로 유지한다.

역할:

- `CModel*` 바인딩
- `AnimationName -> AnimationIndex` 캐싱
- `_uint64 key -> AnimationIndex` 매핑
- `Play(key)`
- `Update(delta)`에서 `CModel::Play_Animation(delta)` 호출
- 현재 클립, finished 여부 관리

하지 않는 일:

- `IDLE`, `DASH`, `ATTACK` 같은 게임 액션 의미를 알지 않는다.
- Character/Monster 구분을 알지 않는다.
- 자동 Idle 복귀를 직접 처리하지 않는다.
- 우선순위나 상태 전이 규칙을 직접 판단하지 않는다.

`CAnimController`의 재생 키에는 모델명을 넣지 않는다.

이유:

- 컨트롤러는 이미 `Bind_Model(m_pModelCom)`으로 특정 `CModel` 인스턴스에 묶인다.
- 키에 모델명까지 넣으면 바인딩된 모델과 키의 모델 식별자가 불일치할 수 있다.
- 모델/애니메이션 세트 선택은 Client의 테이블 선택 단계에서 처리하는 것이 더 명확하다.

## 4. Character / Monster 용어 결정

`Player` 대신 `Character`를 사용하기로 했다.

정의:

- `Character`: 플레이 가능한 캐릭터 계열. 직접 조작 대상일 수도 있고, 팀 파이트 동료 AI일 수도 있다.
- `Monster`: 적/몬스터 계열.
- `ControlType`: 같은 Character를 누가 조작하는지 구분한다.

예:

```cpp
enum class CHARACTER_CONTROL_TYPE : _uint
{
	PLAYER,
	ALLY_AI,
	NPC_AI,

	END
};
```

따라서 `CharacterAnimTable`은 조작 주체와 무관하게 재사용 가능하다.

예:

- 성진우를 직접 조작할 때도 `CharacterAnimTable` 사용
- 성진우를 팀 파이트 동료 AI로 사용할 때도 같은 `CharacterAnimTable` 사용
- 차해인, 백윤호 등 플레이 가능 캐릭터도 같은 구조 사용

## 5. Client 파일 분리 방향

공용 정의 파일은 다음처럼 나눈다.

```txt
Client/Public/Client_Enum.h
Client/Public/Client_Struct.h
Client/Public/Client_Func.h

Client/Public/CharacterAnimTable.h
Client/Private/CharacterAnimTable.cpp

// 추후 Monster 구현 시
Client/Public/MonsterAnimTable.h
Client/Private/MonsterAnimTable.cpp
```

역할:

- `Client_Enum.h`: enum만 정의한다.
- `Client_Struct.h`: 구조체 형태만 정의한다.
- `Client_Func.h`: 상태를 가지지 않는 inline helper 함수만 정의한다.
- `CharacterAnimTable.cpp`: 실제 `static const` 애니메이션 매핑 테이블을 정의한다.
- `CharacterAnimTable.h`: 테이블 접근 함수만 선언한다.

`Client_Struct.h`에는 `Make_CharacterAnimKey()` 같은 함수를 두지 않는다.

이유:

- `Client_Struct.h`는 구조체의 구조를 보여주는 용도다.
- 키 생성 함수는 구조체가 아니라 변환/생성 helper이다.
- 따라서 `Client_Func.h`에 분리하는 것이 더 일관적이다.

설계 이유:

- `BaseAttack_01`, `BaseAttack_01_Fighter`를 별도 Action으로 만들지 않는다.

```txt

BASIC_ATTACK_01 + FIGHTER -> BaseAttack_01_Fighter
SKILL_01 + UNARMED -> Skill_01
SKILL_01 + FIGHTER -> Skill_01_Fighter
SKILL_01 + ONE_HAND_SWORD -> Skill_01_Sword
```

## 8. Client_Func.h 설계안

`Client_Func.h`는 상태 없는 inline helper 함수만 둔다.

```cpp
#ifndef Client_Func_h__
#define Client_Func_h__

#include "Engine_Defines.h"
#include "Client_Enum.h"

namespace Client
{
	inline _uint64 Make_CharacterAnimKey(CHARACTER_ACTION eAction, CHARACTER_WEAPON_STATE eWeapon)
	{
		return (static_cast<_uint64>(eAction) << 32) |
			static_cast<_uint64>(eWeapon);
	}

	inline _uint64 Make_MonsterAnimKey(MONSTER_ACTION eAction, MONSTER_PHASE ePhase)
	{
		return (static_cast<_uint64>(eAction) << 32) |
			static_cast<_uint64>(ePhase);
	}
}

#endif // Client_Func_h__
```

주의:

- `Client_Func.h`에 gameplay 로직, 상태 전이, 컴포넌트 등록 로직을 넣지 않는다.
- 공용 변환/생성 helper만 둔다.

## 10. Body_Player 적용 흐름

`Body_Player::Initialize()` 또는 별도 준비 함수에서 다음 순서로 처리한다.

```cpp
const CHARACTER_ANIM_TABLE_DESC* pTable =
	Find_CharacterAnimTable(CHARACTER_ANIM_SET::SUNGJINWOO_ERANK);

if (nullptr == pTable)
	return E_FAIL;

m_pModelCom->Set_RootBoneName(pTable->pRootBoneName);

if (FAILED(m_pAnimController->Bind_Model(m_pModelCom)))
	return E_FAIL;

for (_uint i = 0; i < pTable->iNumClips; ++i)
{
	const CHARACTER_ANIM_BIND_DESC& Clip = pTable->pClips[i];

	CAnimController::ANIM_CLIP_DESC Desc{};
	Desc.pAnimationName = Clip.pAnimationName;
	Desc.bRestartOnEnter = Clip.bRestartOnEnter;

	if (FAILED(m_pAnimController->Register_Clip(
		Make_CharacterAnimKey(Clip.eAction, Clip.eWeapon),
		Desc)))
		return E_FAIL;
}

m_pAnimController->Play(
	Make_CharacterAnimKey(CHARACTER_ACTION::IDLE, CHARACTER_WEAPON_STATE::COMMON));
```

나중에는 직접 `Make_CharacterAnimKey()`를 매번 호출하지 않고 `Play_Action()` 같은 래퍼로 감싸는 것이 좋다.

예:

```cpp
HRESULT CBody_Player::Play_Action(CHARACTER_ACTION eAction)
{
	// 1. 현재 weapon state 기준 key 생성
	// 2. 없으면 COMMON fallback
	// 3. policy/priority 검사
	// 4. AnimController::Play(key)
}
```

## 13. 보류/추후 검토

- `Capture_RootMotionBaseline()` 추가 여부
	- 첫 프레임 Root Motion delta 손실을 줄이려면 나중에 검토 필요.
	- 전환 직후 track 0 pose를 먼저 적용하고 baseline을 잡는 방식이 이상적이다.

- `MonsterAnimTable` 구체화
	- 실제 몬스터 모델/애니메이션 목록이 들어온 뒤 진행한다.
	- Monster는 `Action + Phase` 축이 더 자연스럽다.

- `Character` 추상 클래스/공통 상태 레이어
	- 현재는 `Body_Player`에 먼저 적용한다.
	- 이후 플레이 가능 캐릭터가 늘어나면 `CCharacter` 계층으로 끌어올릴 수 있다.

- ShowRoom/Win/Fail/Tag_In 계열
	- 전투용 `CHARACTER_ACTION`에 처음부터 넣지 않는다.
	- 로비/결과/연출 상태가 필요할 때 별도 액션으로 추가한다.
