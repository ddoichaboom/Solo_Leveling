## 5-D: 모델 피킹 — 구현 완료

### 문제 분석 및 해결 과정

1. **기존 문제**: CModelObject의 `m_pVIBufferCom = nullptr` → Panel_Viewport 피킹 루프에서 모델 오브젝트 무시됨
2. **1차 해결**: CModel::Pick() 추가 + Panel_Viewport에서 Com_Model 검색 경로 추가
   - NONANIM 모델: 정상 피킹 확인
   - ANIM 모델: 피킹 실패 — PICK_DATA가 바인드 포즈(T-포즈) 정점을 갖고 있어 스키닝된 렌더 결과와 불일치
3. **2차 해결 (CPU 스키닝 피킹)**: 클릭 시점에 현재 본 행렬로 CPU 스키닝한 임시 정점 배열을 만들어 삼각형 테스트
   - CMesh에 정점별 BlendIndex/BlendWeight 보존 (Ready_AnimMesh에서 저장)
   - CModel::Pick() — ANIM일 때 메시별 본 행렬 계산 → CPU 스키닝 → Ray vs 삼각형 → 임시 배열 해제
   - PICK_DATA 원본은 건드리지 않음

### 설계 결정사항

- **CPU 스키닝 방식 채택**: 상용 엔진 조사 결과 바운딩 박스(1차 필터) + CPU 스키닝(정밀)이 일반적. 본 프로젝트는 CPU 스키닝 단독으로 충분 (클릭 시 1회만 실행, 성능 무시 가능)
- **블렌드 데이터 보존 위치**: PICK_DATA 확장 대신 CMesh에 별도 vector 멤버로 저장 — 다른 VIBuffer 파생 클래스(Terrain, Rect)에 영향 없음
- **NONANIM/ANIM 분기**: CModel::Pick() 내부에서 모델 타입별 분기. NONANIM은 기존 CVIBuffer::Pick() 그대로, ANIM만 CPU 스키닝 경로

## 5-C: 애니메이션 전환 보간 (CrossFade 블렌딩) — 구체화 완료, 적용 전

### 설계 결정사항

- **보간 방식**: B안 채택 — SQT 분해 후 보간 (업계 표준). Scale/Translation → Lerp, Rotation(Quaternion) → Slerp
- **단계적 확장**: B안(SQT CrossFade, 저수준) 선행 → C안(Animation State Machine, 고수준) 이후 위에 얹는 구조
- **CrossFade 흐름**: 두 애니메이션 동시 재생, ratio = elapsed / duration (0→1), 본별 SQT 블렌딩
- **prev 루프 정책**: 블렌딩 중 prev는 loop=true 전달 — 짧은 블렌딩 구간에서 갑자기 멈추는 것 방지

### 구체화 진행 현황

| 단계 | 상태 | 내용 |
|------|------|------|
| 근거 (Why) | ✅ | 즉시 전환(하드 컷) 시 애니메이션 끊김. 게임플레이 필수 |
| 이유 (Rationale) | ✅ | B안(SQT) + C안(State Machine) 단계적 확장 |
| 개념 (Concept) | ✅ | Channel이 SQT를 외부로 노출 → 블렌딩 후 Bone에 세팅 |

### 세부 구현 — 코드 제시 완료 (사용자 적용 전)

#### Channel.h/cpp — Get_SQT (구현 완료, 커밋됨)

- `Get_SQT(_float, _uint*, _float3&, _float4&, _float3&)` — Update_TransformationMatrix와 동일한 키프레임 보간 로직이나, Bone에 세팅하지 않고 SQT 값만 출력

#### Animation.h/cpp — Update_SQT (코드 제시, 적용 전)

- `Update_SQT(vector<_float3>&, vector<_float4>&, vector<_float3>&, _float, _bool)` — TrackPosition 전진 + 각 Channel::Get_SQT → 본 인덱스 슬롯에 SQT 기록
- Update_TransformationMatrix와 TrackPosition 전진 로직 중복 (6줄) — 함수 합치기 vs 분리 논의 후 **분리 유지** 결정 (기존 코드 수정 최소화, 단일 책임)

#### Model.h — 블렌딩 멤버 추가 (코드 제시, 적용 전)

- `m_iPrevAnimIndex`, `m_fBlendElapsed`, `m_fBlendDuration(0.2f)`, `m_isBlending(false)`
- getter/setter: `Get_IsBlending`, `Get_BlendDuration`, `Get_BlendRatio`, `Set_BlendDuration`

#### Model.cpp — Play_Animation 블렌딩 분기 (코드 제시, 적용 전)

- `m_isBlending == true`: SQT 버퍼 할당 → prev/curr 양쪽 Update_SQT → 본별 Lerp/Slerp → AffineTransformation → Set_TransformationMatrix
- `fRatio >= 1.f`: 블렌드 완료, prev Reset, 정상 경로 전환
- Combined 행렬 갱신은 블렌딩/일반 공통 (마지막에 1회)

#### Model.cpp — Set_AnimationIndex 블렌드 트리거 (코드 제시, 적용 전)

- `m_fBlendDuration > 0.f`: 블렌드 시작 (prev 보존, curr Reset). 이전 애니메이션 현재 위치에서 계속
- `m_fBlendDuration == 0.f`: 기존 동작 (즉시 전환)

#### Panel_Inspector.cpp — 블렌딩 UI (코드 제시, 적용 전)

- `SliderFloat("Blend Duration", 0~1초)` — 블렌드 시간 조절
- 블렌딩 중: ProgressBar(ratio) + "Blending..." 노란색 텍스트

### 미해결 이슈: 루트 모션과 애니메이션 전환 위치 동기화

**문제 구조:**
- 본 애니메이션은 로컬 공간에서 동작 (셰이더: `FinalPos = Vertex × BoneMatrix(로컬) × WorldMatrix(CTransform)`)
- 루트 본에 Translation 키프레임이 있으면 메쉬가 로컬 공간에서 이동하지만, CTransform의 월드 위치는 갱신되지 않음
- 애니메이션 전환 시 새 애니메이션의 루트 본이 시작 위치(원점)로 돌아가면서 메쉬가 스냅백됨

**발생 시나리오:**
```
1. Dash 애니메이션 재생 → 루트 본 Translation (0,0,0) → (0,0,5)
   → 메쉬 시각적으로 전진, CTransform 위치 불변
2. 다음 애니메이션 전환 → 새 루트 본 시작 위치 (0,0,0)
   → 메쉬가 CTransform 원점으로 순간이동 (스냅백)
```

**해결책 — 루트 모션 추출 (상용 엔진 Unity/Unreal 방식):**
- 매 프레임 루트 본의 Translation delta 추출 → CTransform에 적용 (월드 위치 실제 이동)
- 루트 본 Translation은 원점으로 고정 → 메쉬가 항상 CTransform 중심에 위치
- 애니메이션 전환 시 루트 본이 항상 원점이므로 스냅백 없음, 실제 이동은 CTransform에 누적

**판단 기준:**
- CTransform 기반 이동 (게임 코드에서 Go_Forward 등 호출) → 현재 구현 그대로 문제 없음
- 루트 본 Translation 키프레임이 있는 애니메이션 → 루트 모션 추출 시스템 필요
- → **다음 세션에서 현재 애니메이션의 루트 본 Translation 유무 확인 후 결정**
- 루트 모션 추출은 5-C CrossFade와 별개 기능이므로 독립적으로 구현 가능
