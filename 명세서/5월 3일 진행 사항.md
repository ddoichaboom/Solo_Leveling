## 2. Step 10 — Layer 조작 API (Engine)

### 2.1 설계 의사결정

**컨테이너 유지**: `CLayer::m_GameObjects` 는 `list<CGameObject*>` 그대로. vector 로 바꾸는 안은 기각:

- list 는 중간 erase/insert 가 O(1) + iterator 무효화 없음. Hierarchy DnD 는 빈번한 중간 삽입/삭제가 핵심 시나리오.
- vector 로 전환하면 Engine 의 모든 GameObject 순회 경로가 영향받음. Editor 기능 하나 위해 짊어질 리스크 아님.
- 인덱스 기반 Reorder 가 list 에서 불편하지만 `std::list::splice` 로 같은 list 내 노드 이동이 O(1) (메모리 이동 없음). 인덱스→iterator 변환만 O(N).

**계층화 — low-level primitive + high-level operation**:

- `CLayer::Detach_GameObject` (erase only, **no Release**)
- `CLayer::Attach_GameObject` (push_back, **no AddRef**)
- `CLayer::Reorder_GameObject` (splice 기반)

위 3개를 토대로:

- `CObject_Manager::Move_GameObject` = Detach → Attach (RefCount 보존)
- `CObject_Manager::Reorder_GameObject` = layer 위임
- `CObject_Manager::Remove_GameObject` = Detach + Safe_Release

Move/Remove 차이가 "Release 호출 유무 하나"로 줄어 의도가 명료. Move 도중 RefCount 0 이 되는 순간 없음.

### 2.2 호출 시점 규약

mutation 은 항상 **ImGui Render 단계** (Update 루프 외부) 에서 호출:

- Update 도중 erase 하면 진행 중 iterator 무효화 → 크래시.
- Render 단계는 그 프레임 입력 처리가 끝난 뒤이므로 list 변경 안전.
- 별도 deferred command queue 추상화는 **불필요** — Editor 는 단일 스레드 + UI 콜백이 항상 Render 단계 보장.

### 2.3 GameInstance 위임

기존 패턴 (Add_GameObject / Get_Layers) 과 일치하게 GameInstance 에 위임 함수 3개 추가. 시그니처에서 `class CGameObject*` 인라인 키워드 사용 (Add_RenderGroup 과 동일 스타일) — 헤더 변경 최소.

## 3. Step 11 — Hierarchy 패널 인스턴스 편집

### 3.1 추가 기능

선택은 이미 작동 중이라 신규는 DnD + 삭제 두 가지.

**DnD 통합 규약**:
- 레이어 노드(TreeNode) 드롭 → "이 레이어로 이동, 끝에 append"
- 개별 아이템(Selectable) 드롭 → "이 아이템 *바로 앞에* 삽입"
  - 같은 레이어 → Reorder
  - 다른 레이어 → Move + 그 인덱스로 Reorder

UE/Unity 와 같은 멘탈 모델 ("내가 놓는 위치에 들어간다").

**삭제**: Delete 키 + 우클릭 컨텍스트 메뉴 둘 다 제공.

### 3.2 안전성 처리

**Iterator 무효화 회피**: ImGui Render 도중 mutation 이 발생하면 list iterator 가 깨짐. 단일 `m_PendingCmd` 멤버에 한 건만 저장 후 `ImGui::End` 직후 `Flush_PendingCommand` 에서 처리.

DnD 가 한 프레임에 한 번만 발생한다는 가정 → 단일 슬롯으로 충분.

**DnD payload**:

```cpp
struct DND_OBJECT
{
    CGameObject*    pObject;
    const _wstring* pSrcLayer;   // map key 의 영속 참조 (한 프레임 lifetime 내 안전)
};
```

레벨은 payload 에 안 넣음 — Editor 는 항상 현재 레벨만 다룸.

### 3.3 Remove 후 처리

- 호출자가 보유한 raw 포인터는 즉시 dangling
- 호출 직전 `Panel_Manager::Get_SelectedObject() == 대상` 이면 `Clear_Selection`
- → Inspector 의 dangling 회피

## 5. Step 12 — ContentBrowser Prototype 카탈로그

### 5.1 UI 배치 결정 (TabBar)

기존 ContentBrowser 는 **파일시스템 브라우저** (Resources/ + FBX→bin 변환). Prototype 카탈로그는 다른 종류의 콘텐츠.

세 가지 안 검토 후 **TabBar (한 패널 두 탭)** 선택:

- 둘 다 "스폰/배치할 콘텐츠 소스" — 같은 카테고리.
- 별도 패널 신설은 Panel_Manager / EditorApp 등록 부담 → 이번 Step 범위 초과.
- 좌우 분할은 항상 둘 다 표시되어 화면 낭비.

```
┌─ Content Browser ──────────────────┐
│ [ Files ] [ Prototypes ]           │
├────────────────────────────────────┤
│  Files 탭     → 기존 코드 그대로   │
│  Prototypes 탭 → 신규              │
└────────────────────────────────────┘
```

기존 `Render` 본문을 `Render_Files()` 로 추출. `Render_Prototypes()` 신규.

### 5.2 Prototype 노출 정책

- **GAMEOBJECT 만** 필터링 (Component prototype 은 단독 spawn 불가)
- 그룹: `[STATIC]` (level 0) + `[Level N]` (현재 레벨, level 0 이 아닐 때만)
- 캐싱 없이 매 프레임 `Enum_Prototypes` 호출 — Hierarchy 가 매 프레임 `Get_Layers` 호출하는 패턴과 일관

### 5.3 DnD payload — 안정성 확보

**최초 설계 (실패)**:

```cpp
struct DND_PROTOTYPE
{
    _uint           iLevel;
    const _wstring* pTag;   // ← 지역 vector 의 wstring 주소
};
```

ImGui 가 byte 만 deep copy. 함수 종료 시 지역 vector 파괴 → `pTag` dangling. 다음 프레임 accept 시 `length_error: string too long` 크래시.

**수정 — fixed-size buffer 임베드**:

```cpp
struct DND_PROTOTYPE
{
    _uint   iLevel;
    wchar_t szTag[MAX_PATH];   // payload 자체에 데이터 포함
};
```

ImGui 가 struct 전체를 byte 복사 → tag 데이터까지 보관. lifetime 문제 소멸. 크기 약 524바이트지만 한 프레임 한 번이라 무시 가능.

ContentBrowser 와 Hierarchy 양쪽이 byte 호환이어야 하므로 anonymous namespace 에 동일 정의 + 주석 (`Panel_ContentBrowser.cpp 와 동일해야 함`). 추가 사용처 생기면 그때 공용 헤더로 승격.

## 6. Step 13 — Hierarchy 레이어에 Prototype 드롭 spawn

### 6.1 범위 축소 결정

원래 계획: Viewport 드롭 → Ray Pick → 월드 좌표 spawn.

**보류한 이유**:
- Ray pick + 카메라 좌표 변환 등 부가 작업 부담
- Viewport 드롭 시 "어느 레이어로 넣을지" 결정 부담
- "내가 본 위치에 정확히 놓는다"는 욕구는 Step 14 이후 ImGuizmo 로 위치 수정하면 됨 → 0,0,0 spawn + Gizmo 이동이 거의 같은 UX

**레벨 전환도 보류**:
- 편집 가능한 레벨이 GAMEPLAY 하나뿐 (LOGO/LOADING 은 UI 셸)
- 레벨 = CLevel 파생 C++ 클래스 (외부 scene 파일 없음)
- 직렬화 인프라 없는 상태에서 레벨 전환 시 작업 손실
- → 둘째 편집 가능한 레벨이 생기는 시점에 함께 도입

### 6.2 구현

`CMD_TYPE::SPAWN` 추가:

```cpp
struct PENDING_CMD
{
    CMD_TYPE     eType;
    CGameObject* pObject;
    _wstring     strSrcLayer;
    _wstring     strDstLayer;
    _uint        iInsertIndex;

    // SPAWN 전용
    _uint        iSpawnProtoLevel;
    _wstring     strSpawnProtoTag;
};
```

Hierarchy 의 레이어 TreeNode 의 `BeginDragDropTarget` 블록 안에서 `DND_HIER_OBJ` 와 `DND_PROTOTYPE` 둘 다 accept. SPAWN 케이스는 `Add_GameObject(protoLevel, protoTag, curLevel, layerTag, nullptr)` 호출.

### 6.3 발견 이슈 — pArg=nullptr 비호환 prototype

`CCamera_Free::Initialize` 가 `static_cast<CAMERA_FREE_DESC*>(pArg)` 후 무조건 dereference → nullptr 크래시.

**즉각 수정 (옵션 A — Camera_Free 에 nullptr 가드)**:

```cpp
HRESULT CCamera_Free::Initialize(void* pArg)
{
    m_strName = TEXT("Camera_");
    m_strTag = TEXT("Free");

    if (nullptr == pArg)
        return E_FAIL;

    auto pDesc = static_cast<CAMERA_FREE_DESC*>(pArg);
    m_fMouseSensor = pDesc->fMouseSensor;
    ...
}
```

이제 Camera_Free 카탈로그 드롭 시 fail Log 출력 후 정상 종료.

**향후 검토**:
- 옵션 B: PROTOTYPE_INFO 에 "DESC 필수 여부" 메타데이터 추가 → 카탈로그에서 disabled
- 옵션 C: 각 prototype 이 DEFAULT_DESC 자체 보유 → "드롭 = 무조건 spawn"

### 6.4 발견 이슈 — Camera_Follow target dangling

Hierarchy 에서 Player 삭제 후 prototype 으로 재 spawn → Camera_Follow 전환 시 화면 깨짐.

**원인**: `Level_GamePlay::Ready_Layer_Camera` 에서 Player transform WorldMatrix 주소를 `CAMERA_FOLLOW_DESC::pTargetWorldMatrix` 에 박음 → `CSpringArm::m_pTargetWorldMatrix` 가 raw 포인터를 영구 보관 → Player 삭제 시 dangling, 새 Player 는 다른 주소라 자동 재바인딩 안 됨.

**판단**: Step 13 의 결함이 아닌 Camera_Follow 의 설계 의존성 문제. 별도 단계로 미루고 임시 운용 규칙으로 해결.

**임시 운용 규칙**: Editor 에서 Player 는 delete 금지.

**향후 옵션**:
- (B) Camera_Follow Late_Update 에서 매 프레임 GameInstance 로부터 target lookup → 자동 재바인딩
- (C) Inspector 에서 카메라 target 재바인딩 UI
- (D) GameObject ID 기반 약한 참조 인프라 (가장 견고)
