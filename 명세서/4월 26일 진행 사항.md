## 2. `CTransform_3D::Rotate_Toward_XZ` 신설

### 2.1 동기

기존 `Apply_MoveIntent` 내부에 약 60줄의 회전 보간 로직(현재 yaw → 목표 yaw, 최단 경로 정규화, rad/s 클램프, RIGHT/UP/LOOK 재구성) 이 인라인. Dash 진입 시 "즉시 회전" 도 같은 수학을 필요로 했다 → 두 곳에서 코드 중복 발생 위험.

### 2.2 결정

- **Engine 측 (`CTransform_3D`) 에 두기로 확정**
- 회전 한도 파라미터(`fMaxRadiansThisFrame`) 로 보간/즉시 양쪽 커버
  - 이동 회전: `RotationPerSec * fTimeDelta` 전달 → 보간
  - 즉시 회전: `XM_PI` 전달 → 한 프레임에 어떤 각도라도 도달
- XZ 평면 한정 (Y 축 회전, Pitch/Roll 미변경) — 캐릭터 root 회전 전용 의미 명시

### 2.3 시그니처

```cpp
// Engine/Public/Transform_3D.h
void Rotate_Toward_XZ(_fvector vDirWorld, _float fMaxRadiansThisFrame);
```

### 2.4 핵심 로직 (Engine/Private/Transform_3D.cpp)

1. `vDirWorld` XZ 길이 검사 → 0 이면 no-op
2. `atan2f(dirX, dirZ)` 로 목표 yaw, 현재 LOOK 으로 현재 yaw
3. `fDiff` `[-π, π]` 정규화 (최단 경로)
4. `fMaxRadiansThisFrame` 로 클램프
5. 기존 RIGHT/UP/LOOK **scale 길이 보존** 후 새 yaw 로 RIGHT/LOOK 재구성, UP 은 (0,1,0)

## 3. `CPlayer::Face_DirectionImmediately` 신설

```cpp
void CPlayer::Face_DirectionImmediately(const _float3& vDirWorld)
{
    if (nullptr == m_pTransformCom) return;
    static_cast<CTransform_3D*>(m_pTransformCom)
        ->Rotate_Toward_XZ(XMLoadFloat3(&vDirWorld), XM_PI);
}
```

**위치 결정**: `CPlayer` 측에 두는 것으로 확정. 이유:
- 의미가 "이동 입력 방향으로 즉시 facing 정렬" — 캐릭터 게임플레이 문맥
- Transform 은 **수학 primitive (`Rotate_Toward_XZ`)** 만 제공, **언제 호출할지 (Dash 진입 직전)** 의 정책은 Player/StateMachine 책임이라는 분리 유지

### 4.3 4/25 §3 잔존 이슈 부분 해소 검토

**4/25 §3 원인 요약**:
- `Player::Update` 가 직전 프레임 카메라 yaw 를 읽음 (한 프레임 lag)
- 평소엔 무시 가능, 그러나 카메라 forward ↔ player forward 차이가 **π 근처** 일 때 stale 오차가 증폭됨
- Dash 의 큰 위치 변화가 mismatch 를 카메라에 강하게 남김

**오늘 변경의 효과**:
- 이전 `S + SPACE = BACK_DASH` 케이스가 가장 빈번한 "각도 차 π 근처" 진입 경로였음 (정면 카메라 ↔ 후방 BACK_DASH)
- 새 정책: `S + SPACE` 시 **즉시 180° 회전 후 DASH** → DASH 시작 시점에 player look 이 이미 이동 방향과 정렬됨 → π 차이 자체가 발생하지 않음
- WASD + SPACE 모든 케이스에서 `Face_DirectionImmediately` 가 player look 을 이동 방향과 정렬하므로 stale lag 의 증폭 조건 제거

**여전히 남는 케이스**:
- SPACE 단독 (BACK_DASH) — facing 유지 정책이라 카메라/플레이어 yaw 차이는 입력 시점 그대로 유지됨. 다만 BACK_DASH 는 "정면 바라보기" 가 의도된 동작이므로 § 3 의 "마우스 Y 둔화" 가 체감 버그라기보단 정상 거동에 가까움

→ **잔존 이슈 #4 (Dash 방향 판정) 사실상 해소**. Ch.6 (카메라 stale + basis) 패치는 BACK_DASH 케이스에서 마우스 Y 응답 끊김이 체감되는지 여부에 따라 진행 여부 재판단.

## 7. `Player_StateMachine.cpp` 연산자 우선순위 수정 (Task #5)

### 7.1 원본 식 (수정 전)

```cpp
if ((CHARACTER_ACTION::DASH == eFinished ||
    CHARACTER_ACTION::BACK_DASH == eFinished &&
    true == m_bLastHasMoveIntent))
```

`&&` 가 `||` 보다 우선 → 실제 평가는 `DASH || (BACK_DASH && bLastHasMoveIntent)` 로 됨.

### 7.2 증상

- `DASH` 종료 → WASD 안 눌렸어도 RUN 전이 시도 (§7.4 R-1 위반)
- IDLE/RUN priority 동률 0 → AutoReturn 이 IDLE 로 보낸 직후 RUN 으로 덮어쓰기 → 다음 프레임 IDLE 복귀
- 결과: **한 프레임 RUN 애니메이션 플래시** 가능 (비대칭 거동의 정체)

### 7.3 수정 후

```cpp
const _bool bDashFamily =
    (CHARACTER_ACTION::DASH == eFinished) ||
    (CHARACTER_ACTION::BACK_DASH == eFinished);

if (bDashFamily && true == m_bLastHasMoveIntent)
{
    Try_Transition(ETOUI(CHARACTER_ACTION::RUN));
}
```

비교 결과를 명시 변수로 분리 → 우선순위 함정 회피 + 의도 자가-문서화. 빌드/체감 확인 완료.

## 10. SpringArm Pitch-Yaw 합성 순서 fix (Task #4 재해석)

### 10.1 증상 재정의

4/25 §3 에 "Dash 직후 마우스 Y 둔화" 로 적었던 증상은 **사실 Dash 와 무관** 했음. 별개의 카메라 거동 이상이 누적된 거였고, 사용자가 추가 검토 요청한 후 정확히 재현/특정됨:

- Pitch 끝까지 끌어붙인 상태에서 마우스 X 만 흔들면 화면 상하 각도(Look.y) 가 같이 변함
- 어떤 Yaw 구간(±90° 부근)에서는 마우스 Y 의 실제 효과 범위가 처음 설정한 `[fPitchMin, fPitchMax]` 보다 훨씬 좁아진 느낌
- Yaw ±180° (뒤돌아본 상태) 에서 마우스 Y 가 상하 반전된 것처럼 동작

### 10.2 가설 1 (틀림) — basis Cross degenerate

4/25 §3.2 의 추정: `Camera_Follow::Apply_SpringArmToTransform` 의 `Cross(worldUp, look)` 가 Pitch ±90° 근처에서 degenerate 되어 RIGHT/UP 가 망가짐.

검증용 직교성/단위길이 로깅 (`[BasisBroken]`) 추가 후 실행 → **출력 0건**.

이유: `fPitchMax = 1.0 rad ≈ 57°`, `fPitchMin = -1.4 rad ≈ -80°`. Pitch 가 90° 에 못 가게 클램프되어 있어 Cross 가 항상 정상 직교. 이 가설은 **틀림**.

### 10.3 가설 2 (확정) — Pitch-Yaw 합성 순서

`SpringArm.cpp:108` (수정 전):
```cpp
_vector q = XMQuaternionMultiply(qYaw, qPitch);
```

DirectXMath 의 `XMQuaternionMultiply(A, B)` 는 math 표기로 **`B * A`** (인자 순서 반대 — 함정).

→ math 식: `qPitch * qYaw`. `XMVector3Rotate(v, q)` = `q * v * q⁻¹` 이므로:
- 안쪽: qYaw * v * qYaw⁻¹ — **Yaw 가 먼저** 적용 (월드 Y 축)
- 바깥쪽: qPitch 가 그 결과에 적용 — **Pitch 가 월드 X 축으로** 들어감

`v = (0,0,1)` 적용 결과:
```
Look.x =  cos(Pitch) · sin(Yaw)
Look.y = -sin(Pitch) · cos(Yaw)     ← Yaw 와 결합
Look.z =  cos(Pitch) · cos(Yaw)
```

**검증 로깅** (`[Pitch-Yaw 결합]`, expected = `-sinf(m_fPitch)`, Yaw 무관해야 정상):

| Yaw | Pitch | expected | actual Look.y | actual ÷ expected |
|---|---|---|---|---|
| 0.600 | 0.562 | -0.5329 | -0.4398 | 0.825 ≈ cos(0.600) |
| 0.825 | -1.100 | 0.8912 | 0.6047 | 0.679 ≈ cos(0.825) |
| -1.560 | -1.100 | 0.8912 | 0.0096 | ≈ 0 = cos(-1.56) |
| 3.132 | -1.100 | 0.8912 | -0.8912 | -1 ≈ cos(3.132) |

전 케이스 `Look.y_actual = -sin(Pitch) · cos(Yaw)` 식과 1:1 일치. 가설 확정.

증상이 모두 이 식으로 설명됨:
- Yaw=0 부근: cos(Yaw)≈1 → 정상
- Yaw=±π/2: cos≈0 → Pitch 가 화면에 거의 안 먹힘 (Y 범위 좁아짐)
- Yaw=±π: cos≈-1 → 부호 반전 (마우스 Y 위/아래 뒤집힘)

### 10.4 Fix

`SpringArm.cpp:77` (Initialize) + `SpringArm.cpp:108` (Update_Rotation) 두 곳 동일 패치:

```cpp
_vector q = XMQuaternionMultiply(qPitch, qYaw);   // 기존: (qYaw, qPitch)
```

DirectXMath 인자 순서 swap → math 식 `qYaw * qPitch`:
- 안쪽: qPitch * v * qPitch⁻¹ — Pitch 가 먼저 (v 의 base 프레임 = 월드 → 월드 X 축)
- 바깥쪽: qYaw — 월드 Y 축으로 회전 (Y 좌표 보존)

결과:
```
Look.x =  cos(Pitch) · sin(Yaw)
Look.y = -sin(Pitch)              ← Yaw 무관
Look.z =  cos(Pitch) · cos(Yaw)
```

수정 적용 후 `[Pitch-Yaw 결합]` 로깅 **출력 0건** 확인 → 디버그 블록 제거.

### 10.5 학습 정리 — "좌=월드 / 우=로컬" 규칙의 문맥

4/25 학습 페이지 Ch.4 ("좌측=월드 / 우측=지역") 규칙은 **누적 (accumulation) 맥락** 에서:
- `q_new = Δq * q_old` (math, **좌측 곱**) → Δq 의 축은 **월드** 으로 해석
- `q_new = q_old * Δq` (math, **우측 곱**) → Δq 의 축은 **로컬** 으로 해석

이 규칙은 **여전히 유효**. 다만 이번 케이스는 누적이 아니라 **2-quat 합성** (매 프레임 Yaw·Pitch 두 회전을 처음부터 묶음).

같은 규칙을 합성에도 적용 가능 — 둘 중 하나를 "기존" 으로 간주:

| 코드 | math 식 | 좌 (월드) | 우 (로컬) | Pitch 의 실제 축 | 결과 |
|---|---|---|---|---|---|
| `XMQuaternionMultiply(qYaw, qPitch)` (버그) | `qPitch * qYaw` | qPitch | qYaw | **월드 X** | Yaw 와 결합 |
| `XMQuaternionMultiply(qPitch, qYaw)` (fix) | `qYaw * qPitch` | qYaw | qPitch | **로컬 X (= 카메라 right)** | 분리 ✓ |

Yaw 는 항상 월드 Y 축이 맞고, Pitch 는 카메라가 Yaw 만큼 돈 후의 로컬 right 축이어야 자연스러운 yaw-pitch 카메라가 됨. 4/25 재구성 패치 시점에 합성 순서를 잘못 잡은 게 오늘 드러난 것 — **재구성식 자체는 옳고, 그 안의 좌/우 슬롯 배치가 거꾸로였음**.

### 10.6 함정 (다음에 같은 실수 안 하기)

`XMQuaternionMultiply(A, B)` → math `B * A`. **DirectXMath 호출 시 첫 인자가 math 의 오른쪽 슬롯 (= 로컬 슬롯)**.

판단할 때는 항상:
1. DirectXMath 호출을 math 표기로 펼친 후
2. `q * v * q⁻¹` 의 안쪽부터 바깥쪽 순으로 v 가 회전되는 순서를 확인
3. 좌/우 슬롯 = 월드/로컬 규칙 적용
