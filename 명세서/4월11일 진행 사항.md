## Layer 1 Step 5-C 최종 보류 결정

### 결정 내용

1차 구현(CrossFade SQT 동시 재생 Lerp) 방식을 전부 제거하고 **하드 컷(즉시 전환)** 상태로 복원. 재시도는 플레이어/무기/애니메이션 컨트롤러 구현 시점으로 이월.

### 제거 범위

- Engine
  - `BONE_SQT` 구조체
  - `CChannel::Get_SQT()`
  - `CAnimation::Update_SQT()`
  - CModel 블렌딩 멤버 전체: `m_iPrevAnimIndex`, `m_fBlendElapsed`, `m_fBlendDuration`, `m_isBlending`, `m_BindPoseSQT`, `m_BlendPrevSQT`, `m_BlendCurrSQT`
  - 블렌딩 관련 getter/setter: `Get_IsBlending`, `Get_BlendDuration`, `Get_BlendRatio`, `Set_BlendDuration`, `Cache_BindPoseSQT`
  - `CModel::Play_Animation` 블렌딩 분기
  - `CModel::Set_AnimationIndex` 블렌드 트리거 분기
- Editor
  - `Panel_Inspector` 블렌딩 UI(`SliderFloat`/`ProgressBar`/"Blending..." 텍스트)

### 보류 배경

1. **비채널 본 정밀도 손실** — `XMMatrixDecompose → XMMatrixAffineTransformation` 라운드트립이 원본 `m_TransformationMatrix`(aiNode mTransformation)를 덮어쓰며 오차 누적
2. **한쪽 채널 본 잔존 행렬** — 블렌드 종료 후에도 잘못된 재구성 행렬이 남음
3. **루트 모션 이중 이동** — 루트 본 Translation 키프레임 애니메이션은 A/B 동시 전진 중 월드 위치가 이중 이동
4. **구조 복잡도 증가** — BlendPrev/Curr SQT, BindPose 캐시, Update_SQT, Dirty Mask 등 유지보수 부담 상승

### 판단 기준

블렌딩은 **"언제 어떻게 전환할 것인가"** 의 고수준 제어(상태머신, 파라미터, 전환 조건)와 떼어놓으면 재작업 가능성이 크다. Solo Leveling 프로젝트는 이후 플레이어/무기/애니메이션 컨트롤러를 구현할 예정이며, 그 시점에 블렌딩 방식(CrossFade / Additive / Masked / Blend Tree)을 상태머신 구조와 **함께** 설계하는 것이 자연스러움.

### 재시도 시 참고사항

- **비채널 본은 절대 decompose/recompose 시키지 말 것** — 원본 `m_TransformationMatrix` 유지
- **"Freeze A → Transition to B[0] → Play B"** 방식 검토 가치 있음 — A 전진 중단 + B는 keyframe[0] 고정, 그 사이만 Lerp. 동시 재생 방식보다 구조 단순
- 상태머신 도입 시 CrossFade는 "전환 엣지의 한 속성"으로 자연스럽게 녹아듦

### 관련 미해결 이슈: 루트 모션 위치 동기화

- 본 애니메이션은 로컬 공간 동작, CTransform 월드 위치는 애니메이션이 갱신 안 함
- 루트 본 Translation 키프레임이 있으면 메쉬만 이동, 전환 시 스냅백
- **해결 방향**: 매 프레임 루트 본 delta → CTransform 적용, 루트 본은 원점 고정 (상용 엔진 Unity/Unreal 방식)
- 블렌딩과 독립된 과제이므로 먼저 해결 가능

### Step 2: 행렬 연동 인터페이스 — `CTransform::Set_WorldMatrix`

- **배경**: ImGuizmo::Manipulate() 는 `float[16]` out 버퍼에 조작 결과를 기록 → CTransform 의 `m_WorldMatrix` 로 반영할 공식 통로 필요
- **대안 비교**:
  - (A) `Set_State × 4` (기존 API) — Right/Up/Look/Position 4행을 각각 추출해 설정. 동작은 동등하지만 호출부가 장황
  - (B) `Set_WorldMatrix(const _float4x4&)` 신규 — `m_WorldMatrix = mat;` 한 줄. 명확한 의도
- **채택**: (B). Engine 측 변경은 이 인라인 setter **한 줄만**.
- 변경 파일: `Engine/Public/Transform.h` — `Get_WorldMatrixPtr()` 아래 인라인 setter 추가

```cpp
void Set_WorldMatrix(const _float4x4& mat)
{
    m_WorldMatrix = mat;
}
```

### Step 3: Viewport 오버레이 연동

- `CPanel_Viewport::Render()` 내 `ImGui::Image()` 직후:
  1. `ImVec2 vImagePos = ImGui::GetCursorScreenPos();` 를 Image 호출 **전** 캐싱 (Image 후에는 커서가 이동)
  2. `ImGuizmo::SetDrawlist();` — 현재 창(Viewport) 의 drawlist 에 그림
  3. `ImGuizmo::SetRect(vImagePos.x, vImagePos.y, iRTWidth, iRTHeight);` — Image 가 차지하는 스크린 공간 지정
- 좌표 정합성: SetRect 영역과 Image 영역이 정확히 일치해야 기즈모 조작이 마우스와 동기화됨

### Step 4: 선택 상태 연동 + 피킹 충돌 방지

- `m_pPanel_Manager->Get_SelectedObject()` → `pTransform->Get_WorldMatrixPtr()` 복사 → `Manipulate(View, Proj, op, mode, worldMatrix)` → `IsUsing()` 시 `Set_WorldMatrix(worldMatrix)`
- View/Proj: `m_pGameInstance->Get_Transform(D3DTS::VIEW)` / `D3DTS::PROJ` 사용 (별도 행렬 추출 불필요, row-major ↔ ImGuizmo 호환)
- **기즈모 ↔ 피킹 충돌 해결**:
  - 문제: Manipulate 를 호출하지 않은 프레임에도 ImGuizmo 내부에서 이전 프레임 기즈모 영역 hit-test 가 남아 `IsOver()` 가 true 를 반환 → 빈 공간 피킹 방해
  - 해결: `_bool bGizmoBlocking = false;` 를 Manipulate 블록 **바깥**에서 선언, `pSelected && pTransform` 브랜치 내부에서만 `IsOver() || IsUsing()` 대입. 피킹 가드는 `!bGizmoBlocking` 로 체크

```cpp
_bool bGizmoBlocking = false;
if (nullptr != pSelected)
{
    CTransform* pTransform = pSelected->Get_Transform();
    if (nullptr != pTransform)
    {
        // View/Proj 행렬
        const _float4x4* pViewMatrix = m_pGameInstance->Get_Transform(D3DTS::VIEW);
        const _float4x4* pProjMatrix = m_pGameInstance->Get_Transform(D3DTS::PROJ);
        _float4x4 worldMatrix = *pTransform->Get_WorldMatrixPtr();

        ImGuizmo::Manipulate(
            reinterpret_cast<const _float*>(pViewMatrix),
            reinterpret_cast<const _float*>(pProjMatrix),
            m_eGizmoOperation, m_eGizmoMode,
            reinterpret_cast<_float*>(&worldMatrix));

        if (ImGuizmo::IsUsing())
            pTransform->Set_WorldMatrix(worldMatrix);

        bGizmoBlocking = ImGuizmo::IsOver() || ImGuizmo::IsUsing();
    }
}

if (ImGui::IsItemHovered() &&
    ImGui::IsMouseClicked(ImGuiMouseButton_Left) &&
    !bGizmoBlocking)
{
    // ... 피킹 ...
}
```

### Step 5: 조작 모드 & 단축키 + Panel_Shortcuts

#### 5-1: 단축키 처리

- `CPanel_Viewport` 에 멤버 추가:

```cpp
#pragma region GIZMO
private:
    ImGuizmo::OPERATION  m_eGizmoOperation = ImGuizmo::TRANSLATE;
    ImGuizmo::MODE       m_eGizmoMode      = ImGuizmo::LOCAL;
#pragma endregion
```

- `Render()` 내 Image 직후, Manipulate 호출 전:

```cpp
if (ImGui::IsWindowFocused() && !ImGui::IsMouseDown(ImGuiMouseButton_Right))
{
    if (ImGui::IsKeyPressed(ImGuiKey_W))      m_eGizmoOperation = ImGuizmo::TRANSLATE;
    else if (ImGui::IsKeyPressed(ImGuiKey_E)) m_eGizmoOperation = ImGuizmo::ROTATE;
    else if (ImGui::IsKeyPressed(ImGuiKey_R)) m_eGizmoOperation = ImGuizmo::SCALE;

    if (ImGui::IsKeyPressed(ImGuiKey_X))
        m_eGizmoMode = (m_eGizmoMode == ImGuizmo::LOCAL) ? ImGuizmo::WORLD : ImGuizmo::LOCAL;
}
```

- **Render() 에 배치한 이유**: `ImGui::IsWindowFocused()` 는 해당 창 `Begin()`/`End()` 스코프 내에서만 유효. Update() 에서는 호출 불가. ImGui 이디엄상 패널 Render() 는 "드로잉만 하는 함수" 가 아니라 "프레임 핸들러" 로 취급됨 (입력 검사 + UI 드로잉 혼재가 정상 패턴)
- **2개 병렬 입력 시스템 공존**:
  - Engine `CInput_Device` → 게임 로직 입력 (3D 씬 조작)
  - ImGui `IsKey*` / `IsMouse*` → 에디터 UI 입력
  - 두 시스템이 독립적으로 동작하며 충돌 없음

#### 5-2: CCamera_Free 와의 키 충돌 해결

- **문제**: CCamera_Free 는 RMB-modal 로 WASD/QE 카메라 조작. 기즈모 W(Translate)/E(Rotate) 와 충돌
- **해결**: 단축키 검사에 `!ImGui::IsMouseDown(ImGuiMouseButton_Right)` 가드 추가 — RMB 를 누르지 않았을 때만 기즈모 단축키 처리. RMB 중에는 CCamera_Free 가 독점

#### 5-3: CPanel_Shortcuts 신규 패널 (컨텍스트 민감 안내)

- **요구**: Viewport 내부가 아닌 **별도 패널**로, RMB 상태에 따라 현재 활성 모드를 강조 표시하고 반대 모드는 회색 요약

- **Render() 동작**:

  - `bCameraMode == true` → 노란색 `[ Camera Control - ACTIVE ]` + WASD/QE/Mouse BulletText, 아래 회색 Gizmo 요약
  - `bCameraMode == false` → 청색 `[ Gizmo - ACTIVE ]` + W/E/R/X BulletText, 아래 회색 Camera 요약

### Step 6: Inspector 양방향 동기화

#### 핵심 통찰: 동기화는 이미 자동 작동

- Inspector RTTR 프로퍼티는 **매 프레임** `Get_Position()` / `Get_Rotation()` / `Get_Scale()` 호출 → `m_WorldMatrix` 에서 즉석 추출
- ImGuizmo::Manipulate 는 **매 프레임** `Get_WorldMatrixPtr()` 로 읽어들임
- 둘 다 `m_WorldMatrix` 를 Single Source of Truth 로 폴링 → 별도 dirty 플래그나 이벤트 브로드캐스트 불필요
- 결과:
  - **Gizmo → Inspector**: 기즈모 드래그 중 Inspector 값 실시간 갱신 (RTTR getter 가 매 프레임 재평가)
  - **Inspector → Gizmo**: Inspector 숫자 편집 시 `Set_Position` → `m_WorldMatrix` 갱신 → 다음 프레임 기즈모 위치 즉시 이동

#### 실제 코드 변경: `BeginDisabled` 가드만 추가

기즈모 드래그 중 Inspector 의 Transform 필드를 시각적으로 비활성화 — 드래그 중 값이 빠르게 변하는 상황에서 사용자가 DragFloat 를 잡으면 충돌하는 것을 방지.

```cpp
void CPanel_Inspector::Render_Transform(CTransform* pTransform)
{
    if (!ImGui::CollapsingHeader("Transform", ImGuiTreeNodeFlags_DefaultOpen))
        return;

    _bool bGizmoUsing = ImGuizmo::IsUsing();

    if (bGizmoUsing)
        ImGui::BeginDisabled();

    rttr::type type = rttr::type::get<CTransform>();

    for (auto& prop : type.get_properties())
    {
        Render_Property(prop, *pTransform);
    }

    if (bGizmoUsing)
        ImGui::EndDisabled();
}
```

#### 알려진 한계 (Layer 2-a 범위 밖, 수정 안 함)

- **Euler 회전 라운드트립 불가역성**: WorldMatrix → Euler 분해는 다중값(multivalued) 해석이 존재. 기즈모 회전 후 Inspector 의 X/Y/Z Degree 표시는 수학적으로 동등한 다른 조합이 나올 수 있음 — Inspector 를 "숫자 미세 조정" 용도로 보는 현 설계에서는 허용
- **Scale 부호 손실**: Scale getter 는 행 길이(`XMVector3Length`) 로 계산 → 항상 양수. 거울 반전(음수 스케일) 표현 불가 — 사용 시나리오에 음수 스케일이 없으므로 무시
- **ImGuizmo SCALE 은 항상 LOCAL**: 사양상 제약. WORLD 모드에서도 Scale 조작 시 Local 축을 사용

## 설계 결정사항

- **Engine API 최소 변경** — Layer 2-a 전체에서 Engine 은 `Set_WorldMatrix` 한 줄만 추가. ImGuizmo 의존성은 Editor 에만 격리 (Assimp/RTTR 격리 원칙과 일관)
- **DirectX 행우선 ↔ ImGuizmo 호환** — `_float4x4` 를 `reinterpret_cast<float*>` 로 넘기면 별도 transpose 없이 그대로 동작 (row-major + pre-multiplication 규약 일치)
- **폴링 기반 양방향 동기화** — Inspector 와 Gizmo 가 독립적으로 `m_WorldMatrix` 를 매 프레임 읽고 쓰는 것만으로 동기화 완결. 이벤트 시스템 불필요
- **RMB-modal 공존** — CCamera_Free 의 RMB-WASD 와 ImGuizmo 단축키는 RMB 상태로 배타적 분리 → 키 매핑 재설계 없이 공존
