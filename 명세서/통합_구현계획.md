> 작성일: 2026-04-13

## 핵심 아키텍처 결정사항 (유지)

| 항목 | 결정 |
|------|------|
| Assimp 격리 | Editor 전용. Engine/Client는 Assimp 무의존 |
| RTTR 격리 | Editor 전용. Engine/Client는 RTTR 무의존 |
| 모델 포맷 | FBX → SLMD v1 바이너리(.bin) → Engine은 .bin만 읽음 |
| 씬 포맷 | JSON(편집/확인) + 바이너리(런타임) |
| 씬 로더 위치 | Client DLL |
| CRT 통일 | 모든 프로젝트 동적 CRT (/MDd Debug, /MD Release) |

## 완료 현황

### Layer 0: 공통 인프라 — 완료

- Engine API 확장 (오브젝트/레이어/컴포넌트 getter)
- RTTR 세팅 + Registration (CTransform, CGameObject)
- Hierarchy / Inspector / Log / Content Browser 패널
- 마우스 피킹 (PICK_DATA, CVIBuffer::Pick, WorldRay, 오브젝트 순회)
- CTransform _float3 접근자 (Position/Rotation/Scale)

### Layer 1: 모델 파이프라인 — 완료 (5-C 블렌딩 보류)

- Engine: CModel/CMesh/CMaterial/CBone/CAnimation/CChannel
- Editor: CModel_Converter (FBX → .bin 변환)
- Content Browser: .fbx 우클릭 → Convert to .bin
- Shader_VtxMesh / Shader_VtxAnimMesh
- Inspector: 모델 정보, 애니메이션 리스트/재생/루프, 모델 피킹 (CPU 스키닝)
- 변환된 .bin: SungJinWoo_ERank, Fiona, ForkLift, Rock, Weapon01

### Layer 2-a: ImGuizmo + 스냅 — 완료

- ImGuizmo 통합 (Step 1~6)
- Ctrl+Drag 스냅 (Translate/Rotate/Scale)
- 단축키 (W/E/R/X), CPanel_Shortcuts
- Inspector 양방향 동기화 (BeginDisabled 가드)
- 기즈모 ↔ 피킹 충돌 해결

### ContainerObject / PartObject 구조 — 완료

- Engine: CContainerObject / CPartObject 기반 클래스
- Client: CPlayer(Container) → CBody_Player(Part, Anim) + CWeapon(Part, Static)
- Editor: Container 내부 PartObject 피킹
- 소켓 본 연결 (Prop_Weapon_Dualwield_01_R)

## 이후 단계

### 단계 1: 맵 로드

**목적:** 게임 맵을 3D 모델로 로드하여 Editor/GameApp에서 렌더링

- `CMapObject` 전용 클래스 생성 (Client DLL)
- NONANIM 단일 모델, 기존 .bin 파이프라인 활용 (FBX → Convert → .bin → CModel)
- 맵은 배경 역할 — 선택/이동 대상이 아님
- 맵 표면 피킹은 NavMesh 정점 배치의 기반이 됨

---

### 단계 2: 애니메이션 관리

**목적:** 루트 모션 반영, 상태 기반 애니메이션 전환, 블렌딩 재설계

**순서:**

| # | 항목 | 내용 |
|---|------|------|
| 1 | **루트 모션** | 매 프레임 루트 본 delta → CTransform에 적용, 루트 본은 원점 고정. 블렌딩과 독립, 선행 가능 |
| 2 | **AnimController (상태머신)** | Action + WeaponState → AnimationName → cached index. gameplay 코드가 index를 직접 모르는 구조 |
| 3 | **블렌딩** | 상태머신 전환 엣지의 속성으로 설계. 비채널 본 원본 유지 원칙. Freeze A → Lerp → Play B 방식 검토 |
| 4 | **Editor 프리뷰 분리** | gameplay 상태머신과 Inspector 프리뷰가 충돌하지 않는 구조 |

**이전 실패 교훈 (1차 CrossFade 시도):**
- 비채널 본까지 decompose/recompose 시 정밀도 오차 누적
- 루트 본 Translation 키프레임 있을 때 이중 이동
- 구조 복잡도(BlendPrev/Curr SQT, BindPose 캐시 등) 과도

---

### 단계 3: Player

**목적:** 플레이어 게임플레이 완성 (상태머신, 무기 상태, 입력 연동)

- 단계 2의 AnimController와 맞물려 진행
- Player 상태머신, 무기 장착/해제, 공격/스킬/피격 등
- 세부 사항은 해당 단계 진입 시 구체화

---

### 단계 4: NavMesh

**목적:** Editor에서 맵 위에 이동 가능 영역(NavMesh) 수동 배치

- **수업 코드 참고**: 9개월차 2일차 `CNavigation`/`CCell`
- **편집 워크플로우**: NavMesh 편집 모드 → 맵 표면 피킹 → 히트 포인트(_float3) 획득 → 정점 등록 → 3개 모이면 삼각형 생성
- **자동 Winding 보정**: 정점 순서 무관, Cross Product 법선의 Y 부호로 +Y(위) 방향 통일
- **인접 관리 / 정점 스냅 / 편집 기능**: 구현 시 구체화
- **저장**: 별도 NavMesh 데이터 파일
- **맵 피킹 성능**: 맵 메시 규모에 따라 브루트포스 or 공간 분할(Octree/BVH) 검토

---

### 단계 5: Font + 2D UI 에디터

**목적:** 텍스트 렌더링 도입, 2D 전용 캔버스에서 UI 오브젝트 배치/편집

**Font 시스템:**
- **수업 코드 참고**: 9개월차 2일차 `CFont_Manager`/`CCustomFont` (DirectXTK SpriteFont)
- Engine에 도입

**2D UI 에디터:**
- Editor에 `CPanel_2DCanvas` 신규 패널
- 게임 해상도 기준 직교 투영 좌표계
- 기존 `CUIObject` / `UIOBJECT_DESC` 기반으로 UI 요소 배치/이동/리사이즈
- UI 요소: 이미지, 텍스트, 버튼/슬롯 등 (필요 시 확장)

**UI 씬 구조:**
```
Level 진입 → Base UI 자동 로드 (HUD)
이벤트 발생 → 오버레이 UI 추가 로드 (인벤토리, 보스 HP 등)
이벤트 종료 → 오버레이 UI 해제
```

**저장:** 3D Scene과 별도 UI Scene 파일로 관리

---

### 단계 6: Scene 저장/로드

**목적:** Editor 배치 결과를 파일로 저장하고 GameApp에서 로드

- 각 단계에서 로드할 대상이 구현되는 시점에 맞춰 점진적으로 구축
- **3D Scene**: 오브젝트 배치 (prototype tag, layer, transform, parts/socket 구성)
- **UI Scene**: UI 오브젝트 배치 (위치, 크기, 텍스처, 렌더 순서)
- **NavMesh 데이터**: 별도 파일
- **포맷**: JSON(편집) → 바이너리(런타임)
- **Client CScene_Loader**: 바이너리 fread → Add_Prototype → Add_GameObject

---
