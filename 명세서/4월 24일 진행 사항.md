### 설계 결정

#### (1) 카메라 Yaw 획득 방식: PipeLine Pull

**Camera 를 직접 참조하지 않고** `CGameInstance::Get_Transform(D3DTS::VIEW)` 로 VIEW 행렬을 조회해 Yaw 만 뽑아 씀.

**대안들과 비교한 이유:**

| 방식 | 커플링 | 지연 | 적합성 |
|------|--------|------|--------|
| **Camera 직접 참조** (Layer 에서 dynamic_cast) | 높음 (Camera_Follow 타입 의존) | 0 | Camera 타입 바뀌면 깨짐 |
| **PipeLine Pull** (`Get_Transform(VIEW)`) | 없음 | 0 프레임 | ⭐ 선택 |
| **Camera Notify Push** (`OnNotify`) | 양방향 (리스너 등록/해제) | 타이밍 의존 | 상태를 이벤트로 포장하는 안티패턴 |

**상태 vs 이벤트 구분:**
- **상태**: 매 프레임 유효한 값 (Yaw, 위치). → Pull 패턴
- **이벤트**: 특정 시점 1회 발생 (ACTION_FINISHED, Shake). → Push 패턴 (기존 NotifyListener)

카메라 Yaw 는 매 프레임 상태 → Pull 이 정석. 매 프레임 Notify 를 쏘는 건 본질적으로 Pull 을 흉내내는 오버헤드.

#### (2) 지연 0 프레임의 근거

```
프레임 N:
  1. Priority_Update (모든 객체)
  2. Update 단계 (Layer 알파벳 순)
     ├─ Layer_BackGround
     ├─ Layer_Camera   ← CCamera::Update 에서 Set_Transform(VIEW, ...)
     └─ Layer_Player   ← Get_Transform(VIEW) 조회 — 이번 프레임 값
  3. PipeLine::Update  ← 역행렬 계산
  4. Late_Update
```

- **알파벳 순 "Camera < Player"** 이므로 Camera 가 이번 프레임 VIEW 를 먼저 씀
- Player 는 `Get_Transform(VIEW)` (**역행렬 아님**) 로 같은 프레임 값 조회
- `Get_Transform_Inverse` 를 쓰면 1프레임 지연 (PipeLine::Update 가 Late_Update 직전에 돌기 때문) — 쓰지 않음

**향후 리스크**: Layer 이름이 바뀌어 알파벳 순이 뒤집히면 순서 의존성 깨짐. 장기 리팩터링 항목:
```cpp
// 더 안전한 패턴 (추후 Engine 리팩터링)
void CCamera::Priority_Update(_float fTimeDelta) { Update_PipeLine(); }
void CCamera::Update(_float fTimeDelta) {}
```
지금은 알파벳 의존 유지.

#### (3) Yaw 추출 공식

VIEW 행렬의 회전부는 카메라 World 회전부의 **transpose**. 따라서 카메라 Look 벡터(월드) = VIEW 행렬의 3번째 **열**:

```
lookX = View._13
lookZ = View._33
Yaw   = atan2(lookX, lookZ)
```

역행렬 계산 불필요.

#### (4) 입력 축 → 월드 방향 회전

- 카메라 Forward(월드 XZ) = `(sin(yaw), cos(yaw))`
- 카메라 Right(월드 XZ)   = `(cos(yaw), -sin(yaw))`
- 입력축 `(ax, az)` (ax=좌우, az=전후) 를 회전:
  ```
  vx = az·sin + ax·cos
  vz = az·cos − ax·sin
  ```
- 대각 입력 시 길이 √2 방지를 위해 정규화

```cpp
#include "IntentResolver.h"

void CIntentResolver::Resolve(const PLAYER_RAW_INPUT_FRAME& Raw, _float fCameraYaw, PLAYER_INTENT_FRAME* pOutIntent)
{
    if (nullptr == pOutIntent)
        return;

    pOutIntent->vMoveAxis.x = Compute_Axis(Raw.bMoveLeftHeld, Raw.bMoveRightHeld);
    pOutIntent->vMoveAxis.y = Compute_Axis(Raw.bMoveBackwardHeld, Raw.bMoveForwardHeld);

    pOutIntent->lLookDeltaX = Raw.lMouseDeltaX;
    pOutIntent->bHasMoveIntent = ((0.f != pOutIntent->vMoveAxis.x) || (0.f != pOutIntent->vMoveAxis.y));

    pOutIntent->bDashRequested = Raw.bDashPressed;

    // 카메라 Yaw 를 적용해 월드 이동 방향 계산 (XZ 평면)
    pOutIntent->vMoveDirWorld = { 0.f, 0.f, 0.f };

    if (true == pOutIntent->bHasMoveIntent)
    {
        const _float fSin = sinf(fCameraYaw);
        const _float fCos = cosf(fCameraYaw);

        const _float ax = pOutIntent->vMoveAxis.x;
        const _float az = pOutIntent->vMoveAxis.y;

        // Forward(XZ) = (sin, cos), Right(XZ) = (cos, -sin)
        _float vx = az * fSin + ax * fCos;
        _float vz = az * fCos - ax * fSin;

        const _float fLenSq = vx * vx + vz * vz;
        if (fLenSq > 0.f)
        {
            const _float fInvLen = 1.f / sqrtf(fLenSq);
            vx *= fInvLen;
            vz *= fInvLen;
        }

        pOutIntent->vMoveDirWorld = { vx, 0.f, vz };
    }
}
```

```cpp
void CPlayer::Apply_MoveIntent(const PLAYER_INTENT_FRAME& Intent, _float fTimeDelta)
{
    if (nullptr != m_pStateMachine)
    {
        const CHARACTER_ACTION eCurrent = m_pStateMachine->Get_CurrentCharacterAction();

        const _bool bIsLocomotion =
            (CHARACTER_ACTION::IDLE == eCurrent) ||
            (CHARACTER_ACTION::WALK == eCurrent) ||
            (CHARACTER_ACTION::RUN == eCurrent);

        if (false == bIsLocomotion)
            return;
    }

    if (false == Intent.bHasMoveIntent)
        return;

    CTransform_3D* pTransform = static_cast<CTransform_3D*>(m_pTransformCom);

    // Base(Transform) × 계수(Player) — Option A. E-3 에서 StateMachine 이 계수 교체
    const _float fSpeed = pTransform->Get_SpeedPerSec() * m_fSpeedCoeff;

    _vector vDir = XMVectorSet(Intent.vMoveDirWorld.x, 0.f, Intent.vMoveDirWorld.z, 0.f);
    _vector vPos = pTransform->Get_State(STATE::POSITION);
    vPos = XMVectorAdd(vPos, XMVectorScale(vDir, fSpeed * fTimeDelta));
    pTransform->Set_State(STATE::POSITION, vPos);
}
```

_float CPlayer::Query_CameraYaw() const
{
    // VIEW 의 회전부는 카메라 World 회전부의 transpose.
    // => 카메라 Look 벡터(월드) = (View._13, View._23, View._33)
    const _float4x4* pView = m_pGameInstance->Get_Transform(D3DTS::VIEW);
    if (nullptr == pView)
        return 0.f;

    return atan2f(pView->_13, pView->_33);
}

## 속도 처리 확정 (Option A: Base × Coefficient)

### 배경
- `CTransform` 에 `Get_SpeedPerSec()` getter 없음. `m_fSpeedPerSec` 필드만 존재
- `PLAYER_DESC : GAMEOBJECT_DESC : TRANSFORM_DESC` 상속 구조 — `fSpeedPerSec` 는 TRANSFORM_DESC 에서 이미 Player 로 전달됨

### 결정
- **Base 속도**: `TRANSFORM_DESC::fSpeedPerSec` (immutable, Transform 이 보유)
- **계수 (Coefficient)**: Player 가 보유, 상태에 따라 교체
- **실제 이동 속도** = `base × coeff`

### 왜 A 인가 (Transform setter 직접 변경 B안 대비)

| 항목 | A. 계수 분리 | B. Transform setter 로 base 변경 |
|------|-------------|-------------------------------|
| 원본 base 보존 | ✅ 유지 | ❌ 잃어버림, 별도 캐싱 필요 |
| RootMotion / Go_Straight 부작용 | 없음 | 같은 필드 공유 → 영향 범위 넓음 |
| 상태 전환 시 복구 | 계수만 스왑 | base 재계산 필요 |
| 코드 복잡도 | 단순 | B 는 결국 A 구조를 흉내내야 함 |
