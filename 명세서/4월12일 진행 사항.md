# 4월 12일 진행 사항

## 개요

Scene Loader와 씬 JSON 구조를 확정하기 전에, 임시 `CModelObject` 만으로는 실제 게임 오브젝트 구조를 판단하기 어렵다고 보고 Player/Body/Weapon 조립 구조를 먼저 검토했다.

오늘의 핵심은 다음 세 가지다.

- `CContainerObject` / `CPartObject` 기반으로 Player를 Body와 Weapon 파츠로 분리
- Editor Viewport에서 Container 내부 PartObject까지 피킹되도록 흐름 점검
- Player 애니메이션 47개를 인덱스 직접 참조가 아니라 `Action + WeaponState` 매핑으로 관리해야 한다는 방향 확정

## 1. Transform 초기화 방식 정리

### 현재 방향

`Initialize()` 내부에서 매번 `m_pTransformCom->Set_Position()` / `Set_Rotation()` / `Set_Scale()` 을 직접 호출하기보다, 초기 Transform 값은 `TRANSFORM_DESC` / `GAMEOBJECT_DESC` 에 담아 `CGameObject::Initialize()` → `CTransform::Initialize()` 흐름에서 한 번에 적용하는 방향이 더 합리적이다.

현재 구조:

```cpp
struct TRANSFORM_DESC
{
    _float3 vPosition = {};
    _float3 vRotationDeg = {};
    _float3 vScale = { 1.f, 1.f, 1.f };

    _float  fSpeedPerSec = {};
    _float  fRotationPerSec = {};
};
```

```cpp
HRESULT CTransform::Initialize(void* pArg)
{
    if (nullptr == pArg)
        return S_OK;

    auto pDesc = static_cast<TRANSFORM_DESC*>(pArg);

    m_fRotationPerSec = pDesc->fRotationPerSec;
    m_fSpeedPerSec = pDesc->fSpeedPerSec;

    Set_Scale(pDesc->vScale);
    Set_Rotation(pDesc->vRotationDeg);
    Set_Position(pDesc->vPosition);

    return S_OK;
}
```

### 판단

- 초기 배치값은 DESC로 전달하는 것이 Scene Loader / JSON 로딩과 잘 맞음
- 런타임 중 이동/회전/스케일 변경은 기존 `Set_Position`, `Set_Rotation`, `Set_Scale`, `Set_State` 사용
- Scene JSON의 `transform` 블록은 그대로 `GAMEOBJECT_DESC` 또는 파생 DESC의 기반 데이터로 매핑 가능

## 2. ContainerObject / PartObject 구조 도입

### 목적

수업 코드의 Container/Part 패턴을 현재 프로젝트에 맞게 이해하고, Player처럼 여러 모델이 조립되는 오브젝트를 Scene Loader 설계 전에 먼저 검증한다.

### 현재 구조

```cpp
class CContainerObject abstract : public CGameObject
{
protected:
    map<const _wstring, class CPartObject*> m_PartObjects;

protected:
    HRESULT Add_PartObject(
        _uint iPrototypeLevelIndex,
        const _wstring& strPrototypeTag,
        const _wstring& strPartTag,
        void* pArg = nullptr);
};
```

```cpp
class CPartObject abstract : public CGameObject
{
public:
    struct PARTOBJECT_DESC : public CGameObject::GAMEOBJECT_DESC
    {
        const _float4x4* pParentMatrix = { nullptr };
    };

protected:
    const _float4x4* m_pParentMatrix = {};
    _float4x4        m_CombinedWorldMatrix = {};
};
```

### 현재 구현된 관계

```text
CPlayer
  ├─ Body   : CBody_Player
  └─ Weapon : CWeapon
```

- `CPlayer` 는 `CContainerObject` 파생
- `CBody_Player` 는 `CPartObject` 파생, 애니메이션 모델을 담당
- `CWeapon` 은 `CPartObject` 파생, Body의 소켓 본에 붙는 무기 모델을 담당
- `CPlayer::Ready_PartObjects()` 에서 Body를 먼저 만들고, Body의 본 포인터를 받아 Weapon을 생성

현재 사용 중인 소켓 본:

```cpp
"Prop_Weapon_Dualwield_01_R"
```

### Scene JSON 설계에 주는 의미

Player 같은 오브젝트는 단순히 모델 하나를 배치하는 구조가 아니다. JSON에서 다음 요소를 표현할 수 있어야 한다.

```json
{
  "object_type": "CPlayer",
  "layer": "Layer_Player",
  "transform": {
    "position": [0, 0, 0],
    "rotation_deg": [0, 0, 0],
    "scale": [1, 1, 1]
  },
  "parts": [
    {
      "tag": "Body",
      "prototype": "Prototype_GameObject_Body_Player"
    },
    {
      "tag": "Weapon_R",
      "prototype": "Prototype_GameObject_Weapon",
      "attach_to_part": "Body",
      "socket_bone": "Prop_Weapon_Dualwield_01_R"
    }
  ]
}
```

다만 1차 Scene Loader에서는 이 구조를 전부 일반화하기보다, Player 전용 DESC/초기화 흐름을 먼저 안정화한 뒤 JSON 일반화를 진행하는 것이 안전하다.

## 4. Viewport Picking / Inspector 확인

### 문제

Player는 ContainerObject이고 실제 렌더링되는 모델은 내부 PartObject인 Body/Weapon이므로, 기존 GameObject 단위 피킹만으로는 Body나 Weapon을 선택하기 어렵다.

### 현재 처리 방향

`Panel_Viewport::Pick_Object()` 에서 다음 순서로 검사한다.

1. 일반 `CGameObject`
   - `Get_VIBuffer()` 가 있으면 VIBuffer 피킹
   - 없으면 `Com_Model` 을 찾아 CModel 피킹
2. `CContainerObject`
   - `Get_PartObjects()` 로 내부 PartObject 순회
   - 각 PartObject의 `Com_Model` 을 찾음
   - `CPartObject::Get_CombinedWorldMatrix()` 기준으로 CModel 피킹
   - 가장 가까운 PartObject를 선택

### 결과

- Body_Player 선택 가능
- Inspector에서 Body_Player의 `Com_Model` 정보 확인 가능
- 애니메이션 목록도 표시됨

## 5. Inspector 애니메이션 미리보기 문제

### 증상

Inspector의 Animation List에서 다른 애니메이션을 클릭해도 실제 재생이 바뀌지 않는 것처럼 보였다.

### 원인

Inspector 클릭 자체는 `CModel::Set_AnimationIndex(i)` 까지 도달한다. 하지만 `CBody_Player::Update()` 에서 Player 상태 기반 애니메이션 전환이 매 프레임 다시 호출되면 Inspector에서 선택한 애니메이션이 바로 덮어써진다.

문제 흐름:

```cpp
void CBody_Player::Update(_float fTimeDelta)
{
    if (*m_pParentState & CPlayer::PLAYER_STATE::IDLE)
        m_pModelCom->Set_AnimationIndex(0);

    if (*m_pParentState & CPlayer::PLAYER_STATE::RUN)
        m_pModelCom->Set_AnimationIndex(1);

    m_pModelCom->Play_Animation(fTimeDelta);
}
```

### 현재 임시 판단

- Inspector 애니메이션 미리보기 테스트 중에는 state-driven `Set_AnimationIndex()` 를 임시로 막아도 됨
- 장기적으로는 Editor Preview 모드와 Gameplay AnimationController를 분리해야 함
- `CModel` 자체가 상태머신을 알면 안 되고, Player/Body 쪽의 AnimationResolver 또는 AnimationController가 재생 결정을 맡아야 함

## 6. 무기 유무에 따른 애니메이션 관리 고민

### 문제

현재 캐릭터 애니메이션은 약 47개이며, Animation List에 단순 인덱스 순서로 나열하면 관리가 어렵다.

애니메이션 성격이 섞여 있다.

- 이동
- 대기
- 공격
- 스킬
- 피격
- 무기 장착 전용
- 무기 없는 상태 전용

특히 `Fighter` 이름이 들어간 애니메이션은 무기 없이 재생되어야 한다.

### 잘못 가기 쉬운 방향

다음처럼 enum 값에 animation index를 직접 박는 방식은 위험하다.

```cpp
enum PLAYER_ANIM
{
    IDLE = 0,
    RUN = 1,
    ATTACK = 17,
};
```

이 방식은 `.bin` 재생성, FBX export 순서 변경, 애니메이션 추가/삭제에 취약하다.

### 권장 방향

Gameplay 코드는 index를 직접 알지 않는다.

```text
Input
  -> PLAYER_ACTION
  -> WEAPON_STATE
  -> AnimationResolver
  -> AnimationName 또는 캐싱된 AnimationIndex
  -> CModel::Set_AnimationIndex()
```

기본 enum:

```cpp
enum class PLAYER_ACTION
{
    IDLE,
    RUN,
    DASH,
    BASIC_ATTACK_01,
    BASIC_ATTACK_02,
    SKILL_01,
    SKILL_02,
    HIT,
    DEATH,
};

enum class WEAPON_STATE
{
    UNARMED,
    DUAL_BLADE,
    DAGGER,
    SWORD,
};
```

매핑 데이터:

```cpp
struct PLAYER_ANIM_DESC
{
    PLAYER_ACTION   eAction = {};
    WEAPON_STATE    eWeaponState = {};
    const char*     pAnimName = nullptr;
    _bool           bLoop = false;
    _bool           bShowWeapon = true;
};
```

예시:

```cpp
static const PLAYER_ANIM_DESC g_PlayerAnimTable[] =
{
    { PLAYER_ACTION::IDLE,            WEAPON_STATE::UNARMED,    "Fighter_Idle",          true,  false },
    { PLAYER_ACTION::IDLE,            WEAPON_STATE::DUAL_BLADE, "Dualwield_Idle",        true,  true  },
    { PLAYER_ACTION::BASIC_ATTACK_01, WEAPON_STATE::UNARMED,    "Fighter_Attack_01",     false, false },
    { PLAYER_ACTION::BASIC_ATTACK_01, WEAPON_STATE::DUAL_BLADE, "Dualwield_Attack_01",   false, true  },
};
```

초기화 시에는 `pAnimName` 을 `CModel::Get_AnimationIndex()` 로 찾아 runtime table에 index를 캐싱한다.

### 무기 표시 처리

애니메이션 이름에 `Fighter` 가 들어갔는지 매 프레임 검사해서 무기를 숨기는 방식은 임시 디버그용으로는 가능하지만, 최종 로직으로는 부적절하다.

최종 방향:

```cpp
struct PLAYER_ANIM_RUNTIME_DESC
{
    PLAYER_ACTION   eAction = {};
    WEAPON_STATE    eWeaponState = {};
    _uint           iAnimIndex = {};
    _bool           bLoop = false;
    _bool           bShowWeapon = true;
};
```

`bShowWeapon` 값으로 Weapon PartObject의 렌더 등록 여부를 제어한다.

```cpp
if (false == m_bVisible)
    return;
```

Weapon을 없애는 것이 아니라, 장착 상태와 애니메이션 상태에 따라 렌더링만 막는 방식이 우선 적절하다.

## 7. Scene Loader / JSON 구조에 반영할 점

### 유지할 방향

이전에 논의한 절충안은 여전히 유효하다.

- A안의 prototype registry 구조를 기본으로 사용
- C안의 `object_type`, `active` 같은 확장 필드 일부 흡수
- Scene Loader 1차는 엔진의 `Add_Prototype` → `Add_GameObject` 흐름에 맞춰 단순하게 구현

### 새로 반영된 고려사항

Player 같은 캐릭터는 다음 데이터를 가진다.

- ContainerObject 여부
- PartObject 목록
- Body 모델
- Weapon 모델
- Weapon이 붙을 Body 파트와 Socket Bone 이름
- 초기 Transform
- 초기 무기 상태

하지만 액션별 애니메이션 매핑은 Scene 배치 데이터와 성격이 다르다.

따라서 분리하는 편이 좋다.

```text
Scene JSON
  - 어떤 오브젝트를 어느 레이어에 배치할 것인가
  - 어떤 prototype을 사용할 것인가
  - transform은 무엇인가
  - part/socket 구성은 무엇인가

Character Animation Data
  - PLAYER_ACTION + WEAPON_STATE 조합
  - AnimationName
  - Loop 여부
  - Weapon 표시 여부
  - 전환/블렌딩 정책
```

### 1차 JSON 예시 방향

```json
{
  "gameobjects": [
    {
      "name": "Player",
      "object_type": "CPlayer",
      "prototype": "Prototype_GameObject_Player",
      "layer": "Layer_Player",
      "active": true,
      "transform": {
        "position": [0, 0, 0],
        "rotation_deg": [0, 0, 0],
        "scale": [1, 1, 1]
      },
      "desc": {
        "initial_weapon_state": "DUAL_BLADE"
      }
    }
  ]
}
```

Player 내부 Body/Weapon 구성이 코드 고정이면 1차 Loader는 이 정도로 충분하다. 이후 프리팹/캐릭터 데이터 분리가 필요해질 때 `parts` 블록을 JSON으로 끌어올린다.

## 오늘의 판단 요약

- Transform 초기값은 DESC 기반으로 전달하는 것이 Scene Loader와 가장 잘 맞음
- `CModelObject` 는 임시 테스트용이고, 실제 캐릭터 구조는 `CPlayer` + `CBody_Player` + `CWeapon` 조립 구조를 기준으로 봐야 함
- PartObject 피킹은 Container 내부까지 순회해야 Editor에서 실사용 가능
- Inspector 애니메이션 미리보기는 Gameplay 상태머신과 분리되어야 함
- 애니메이션은 index가 아니라 `Action + WeaponState -> AnimationName -> cached index` 구조로 관리
- 무기 표시 여부는 애니메이션 이름 문자열이 아니라 명시적 데이터(`bShowWeapon`)로 관리
