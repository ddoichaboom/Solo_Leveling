# 5월 21일 진행 사항 — Igris Boss AI 1차 구현

## 1. 목표

Igris Boss 를 일반 추적형 몬스터가 아니라 보유 애니메이션을 중심으로 움직이는 패턴형 보스로 구성한다.

- CHASE 애니메이션이 없는 자산 조건을 인정하고, 거리 기반 패턴 선택으로 시작한다.
- Boss Encounter 는 정식 Trigger Box 구현 전까지 NavMesh Cell 97 진입으로 임시 발동한다.
- HitBox Notify 배치 전, 현재 AI 가 실제로 사용하는 애니메이션과 RootMotion 정책을 먼저 고정한다.

---

## 2. 적용 설계

### 2.1 Encounter Trigger

- `CBoss_Monster::Initialize()` 에서 즉시 `Begin_Encounter()` 하던 임시 시작은 제거.
- `CBoss_Monster::Update()` 에서 아직 Encounter 전이면 Target 위치를 Boss NavMesh 로 `Find_Cell()` 한다.
- Target 위치가 Cell `97` 이면 `Begin_Encounter()` 호출.
- `Begin_Encounter()` 는 `Intro_A` 재생 후 AI 판단을 시작한다.

정식 작업 전까지의 임시 정책이며, 이후 Trigger Box 또는 SceneData 기반 Trigger 로 교체한다.

### 2.2 거리 기준

현재 테스트 기준:

| 구간 | 거리 |
|---|---:|
| 근거리 | `<= 15.0f` |
| 중거리 | `<= 30.0f` |
| 원거리 | `<= 50.0f` |

AI 판단 주기는 `0.4f` 로 설정했다.

### 2.3 현재 AI 사용 애니메이션

| 용도 | 애니메이션 |
|---|---|
| Encounter | `Intro_A` |
| 기본 상태 | `Battle_Idle` |
| 근거리 | `Skill_04`, `Skill_02`, `Base_Attack_01` |
| 중거리 | `Skill_01`, `Skill_13`, `Skill_05` |
| 원거리 | `Skill_06`, `Skill_01`, `Skill_05` |
| Crash 후 후보 | `Skill_11`, `Skill_10_Start/Loop/End`, `Skill_06`, `Skill_01`, `Skill_13` |
| 상태 전환 | `Crash_Start`, `Crash_Loop`, `Dead_A`, `Dead_Pose_Loop` |

AnimTable 에 등록되어 있지만 현재 AI 가 직접 선택하지 않는 클립:

- `Skill_03`
- `Skill_07_Start/Loop/End`
- `Skill_08_Start/End`
- `Skill_09_Start/Loop/End`
- `Skill_12_Start/Loop/End`
- `Skill_15_Start/Loop/End`

## 3. 패턴별 결정

### Skill_01

- 플레이어 위치로 접근하는 돌진계 패턴.
- 시작 시점의 플레이어 위치를 저장하고, 플레이어 앞 `2.8f` 지점을 목표 위치로 계산한다.
- RootMotion 은 유지하되, Boss 전용 `Apply_RootMotion()` override 에서 X/Z 이동량만 목표 거리 기준으로 스케일링한다.
- 기준 이동량 `10.5f`, scale clamp `0.35f ~ 5.0f`.
- 플레이어를 실시간 추적하지 않으므로 회피 가능한 돌진으로 남긴다.

### Skill_04

- 제자리 점프 공격.
- RootMotion 유지.
- 근거리 후보.

### Skill_06

- 근거리 추적 공격이 아니라 원거리 검기 발사 애니메이션으로 정정.
- 원거리(`> 30.0f && <= 50.0f`) 후보.
- 재생 중 지속 회전하지 않고, 선택 시점 방향만 사용한다.
- 검기 생성은 이후 AnimNotify 시점에 projectile 로 연결한다.

### Skill_10

- 보스 중심 다단 동심원 장판.
- `Start -> Loop -> End` 흐름.
- 제자리 장판이므로 `Skill_10_Start/Loop/End` 모두 RootMotion OFF 로 강제.
- Loop 는 현재 `2.1f` 후 End 로 넘긴다.

### Skill_11

- 보스 중심 단일 원형 장판.
- Crash 후 `Skill_10` 으로 강제 연결하지 않고, 랜덤 후보 중 하나로 둔다.

### Skill_13

- 플레이어 방향 강공격.
- RootMotion 유지.
- 중거리 및 Crash 후 후보.

## 5. RootMotion / Loop 적용 구조

`MONSTER_ANIM_BIND_DESC` 에 애니메이션별 override 필드를 추가했다.

- `bOverrideLoop`
- `bLoop`
- `bOverrideRootMotion`
- `bUseRootMotion`

`CBody_Monster::Register_AnimationClips()` 에서 override 가 켜진 경우에만 `CModel::Set_AnimationLoop()` / `Set_AnimationUseRootMotion()` 을 적용한다.

현재 강제 정책:

- `Skill_10_Start`: Loop OFF, RootMotion OFF
- `Skill_10_Loop`: Loop ON, RootMotion OFF
- `Skill_10_End`: Loop OFF, RootMotion OFF
- `Skill_04`, `Skill_13`: 기존 `.bin` RootMotion 메타 유지

### 이슈 #1 — Kasaka F 클립이 느림 / 어색
- 클립 자체가 SSR(컷씬형) 시퀀스 (도약 + 공중 정지 + 내려찍기)
- 핵심 원인: NavMesh `Try_Move` (`Engine\Private\NavMesh.cpp` line 415/438) 가 매 프레임 `pOutAdjustedPosition->y = Cell->Compute_Height(...)` 로 Y 강제 덮어쓰기 → Engine 의 root motion Y delta (`Model.cpp:906`) 가 즉시 죽음 → 공중 모션이 시각적으로 사라짐
- 해결 제안 (미적용): `CPlayer::Is_AerialAction()` 헬퍼 + `Try_ApplyMovementPosition` 에서 공중 액션이면 NavMesh Y 클램프 우회
- 판단: R6-D (Kasaka 타깃/텔레포트) 와 함께 처리해야 클립 의도가 자연스럽게 정당화됨

### 이슈 #2 — KnightKiller Loop/End 가 콜라이더보다 전방
- Start 는 root motion 정상 동기화
- Loop/End 는 *Start 가 도달한 전방 위치* 를 t=0 기준으로 저작되어 콜라이더에서 재생 시 메시가 전방 오프셋
- 충돌로 자동 정상화 안 됨
- 해결안 3가지:
  - A. Editor Inspector 에서 Loop/End 의 `PreTransform` 음수 Z 오프셋 튜닝 (SLMD v2 가 per-animation PreTransform 지원) — 추천
  - B. Start→Loop 전이 시점에 캐릭터 위치 +Z 스냅
  - C. FBX 재익스포트 (origin 정렬)

### 이슈 #3 — 1.15배 스케일 후 BASIC_ATTACK 슬라이드 (롤백)
- 사용자가 Player 비주얼 + 콜라이더 1.15배 키운 후 발생
- 몬스터 인접 콤보 공격 시 몬스터 원주를 따라 슬라이드
- 원인:
  - `PLAYER_BODY_BLOCK_RADIUS = 0.35f` 하드코딩 (스케일링 영향 없음)
  - `Resolve_BodyBlockingPosition` 의 touch-and-stop tangent 가 매 프레임 미세 회전
  - `Resolve_BodyOverlapPosition` 의 radial push 방향이 매 프레임 변동
- 1차 시도 (롤백): 두 Resolve 에 `Is_AttackAction()` 분기로 하드 스톱 → push 사라져 몬스터와 완전 겹침
