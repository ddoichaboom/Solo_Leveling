# 5월 21일 진행 사항 — Igris Boss AI 1차 구현

## 1. 목표

Igris Boss 를 일반 추적형 몬스터가 아니라 보유 애니메이션을 중심으로 움직이는 패턴형 보스로 구성한다.

- CHASE 애니메이션이 없는 자산 조건을 인정하고, 거리 기반 패턴 선택으로 시작한다.
- Boss Encounter 는 정식 Trigger Box 구현 전까지 NavMesh Cell 97 진입으로 임시 발동한다.
- HitBox Notify 배치 전, 현재 AI 가 실제로 사용하는 애니메이션과 RootMotion 정책을 먼저 고정한다.

---

## 2. 적용 설계

### 2.1 Encounter Trigger

- `CBoss_Monster::Initialize()` 에서 즉시 `Begin_Encounter()` 하던 임시 시작은 제거.
- `CBoss_Monster::Update()` 에서 아직 Encounter 전이면 Target 위치를 Boss NavMesh 로 `Find_Cell()` 한다.
- Target 위치가 Cell `97` 이면 `Begin_Encounter()` 호출.
- `Begin_Encounter()` 는 `Intro_A` 재생 후 AI 판단을 시작한다.

정식 작업 전까지의 임시 정책이며, 이후 Trigger Box 또는 SceneData 기반 Trigger 로 교체한다.

### 2.2 거리 기준

현재 테스트 기준:

| 구간 | 거리 |
|---|---:|
| 근거리 | `<= 15.0f` |
| 중거리 | `<= 30.0f` |
| 원거리 | `<= 50.0f` |

AI 판단 주기는 `0.4f` 로 설정했다.

### 2.3 현재 AI 사용 애니메이션

| 용도 | 애니메이션 |
|---|---|
| Encounter | `Intro_A` |
| 기본 상태 | `Battle_Idle` |
| 근거리 | `Skill_04`, `Skill_02`, `Base_Attack_01` |
| 중거리 | `Skill_01`, `Skill_13`, `Skill_05` |
| 원거리 | `Skill_06`, `Skill_01`, `Skill_05` |
| Crash 후 후보 | `Skill_11`, `Skill_10_Start/Loop/End`, `Skill_06`, `Skill_01`, `Skill_13` |
| 상태 전환 | `Crash_Start`, `Crash_Loop`, `Dead_A`, `Dead_Pose_Loop` |

AnimTable 에 등록되어 있지만 현재 AI 가 직접 선택하지 않는 클립:

- `Skill_03`
- `Skill_07_Start/Loop/End`
- `Skill_08_Start/End`
- `Skill_09_Start/Loop/End`
- `Skill_12_Start/Loop/End`
- `Skill_15_Start/Loop/End`

## 3. 패턴별 결정

### Skill_01

- 플레이어 위치로 접근하는 돌진계 패턴.
- 시작 시점의 플레이어 위치를 저장하고, 플레이어 앞 `2.8f` 지점을 목표 위치로 계산한다.
- RootMotion 은 유지하되, Boss 전용 `Apply_RootMotion()` override 에서 X/Z 이동량만 목표 거리 기준으로 스케일링한다.
- 기준 이동량 `10.5f`, scale clamp `0.35f ~ 5.0f`.
- 플레이어를 실시간 추적하지 않으므로 회피 가능한 돌진으로 남긴다.

### Skill_04

- 제자리 점프 공격.
- RootMotion 유지.
- 근거리 후보.

### Skill_06

- 근거리 추적 공격이 아니라 원거리 검기 발사 애니메이션으로 정정.
- 원거리(`> 30.0f && <= 50.0f`) 후보.
- 재생 중 지속 회전하지 않고, 선택 시점 방향만 사용한다.
- 검기 생성은 이후 AnimNotify 시점에 projectile 로 연결한다.

### Skill_10

- 보스 중심 다단 동심원 장판.
- `Start -> Loop -> End` 흐름.
- 제자리 장판이므로 `Skill_10_Start/Loop/End` 모두 RootMotion OFF 로 강제.
- Loop 는 현재 `2.1f` 후 End 로 넘긴다.
- 2026-05-22 후속 설계:
  - `Skill_10_Start` 끝자락 Notify 는 가장 작은 원의 예고/충전 시작점으로 본다. 바닥면 원이 보스 중심에서 바깥으로 차오르고, 충전 완료 시 1타 판정(`0 ~ R1`)을 낸다.
  - `Skill_10_Loop` 의 첫 번째 Notify 는 2타 링 예고/충전 시작점으로 본다. 충전 완료 시 이전 작은 원을 제외한 영역(`R1 ~ R2`)에 데미지를 준다.
  - `Skill_10_Loop` 의 두 번째 Notify 는 3타 링 예고/충전 시작점으로 본다. 충전 완료 시 두 번째 원을 제외한 외곽 링(`R2 ~ R3`)에 데미지를 준다.
  - 세 번째 원이 거의 다 차오를 때 `Skill_10_End` 로 전환한다. End 구간의 최종 폭발은 전체 원 범위(`0 ~ R3`)에 데미지를 준다.
  - 즉, `ATTACK_HITBOX_ON` 을 항상 즉시 데미지로 해석하지 않는다. `Skill_10` 에서는 단계별 장판 충전 시작 트리거로 쓰고, 내부 타이머/이펙트 fill 완료 시점에 실제 타격을 수행한다.
  - 이펙트는 바닥면 Ground Decal / Ring / Crack 텍스처로 표시하지만, 실제 데미지 판정은 바닥 원이 아니라 보스 중심의 반구체성 판정으로 처리한다.
  - 1차 구현은 진짜 Hemisphere Collider 가 아니라 `XZ ring 범위 + Y 높이 제한` 수학 판정으로 시작한다. 경사/공중 모션을 고려해 플레이어가 약간 떠 있어도 맞되, 너무 높은 위치는 제외한다.
  - 시작 튜닝값 후보: `R1 = 4.5f`, `R2 = 8.0f`, `R3 = 12.0f`, `Height = 4.0f`. 실제 스테이지 스케일과 플레이어 피격 체감에 따라 조정한다.
  - Boss 에도 Player 의 `m_pSkillCollider` 와 같은 공용 스킬 판정 Collider 슬롯을 추가한다. 단, 이것은 모든 스킬을 Collider 로 처리하겠다는 의미가 아니라 `Skill_04`, `Skill_11` 같은 단발 Sphere/OBB 범위 판정과 `Skill_01/05/13` 의 전방 보조 판정용이다.
  - `Skill_10` 은 링/도넛/전체원 구조라 공용 Collider 하나로 표현하기 어렵다. 따라서 `m_pSkillCollider` 대상에서 제외하고, `Apply_HemisphereRingDamage(inner, outer, height, damage)` 같은 수학 판정으로 분리한다.

### Skill_11

- 보스 중심 단일 원형 장판.
- Crash 후 `Skill_10` 으로 강제 연결하지 않고, 랜덤 후보 중 하나로 둔다.

### Skill_13

- 플레이어 방향 강공격.
- RootMotion 유지.
- 중거리 및 Crash 후 후보.

## 4. Igris Boss 이펙트 리소스 활용 메모

원본 Unity 리소스에서 추출한 `MonoBehaviour`, `Material`, `Texture2D`, `Mesh`, `Audio` 정보는 원본 prefab 을 그대로 복원하는 목적이 아니라, 현재 DirectX 프로젝트의 이펙트 시스템에 맞게 **동일 역할의 이펙트 프리셋을 재구성하기 위한 설계 자료**로 사용한다.

| 추출 정보 | 확인할 내용 | 프로젝트 적용 방식 |
|---|---|---|
| `MonoBehaviour` | 이펙트 prefab 이름, 생성 시점, 위치/스케일, projectile 속도, sound 참조 | AnimNotify 타이밍, projectile 생성 파라미터, area spawn 위치 결정 |
| `Material` | 셰이더, main/mask/noise/pattern 텍스처, panner/dissolve 값 | 이펙트 머티리얼 슬롯 구성, UV 흐름, alpha/dissolve 처리 |
| `Texture2D` | 실제 검기, ring, crack, smoke, aura 이미지 | 확정 후보만 `Resources/Textures/Effects/Monster/Igris/` 로 복사 |
| `Mesh` | plane, trail, ring, circle, crack, half sphere 형태 | 월드 projectile, ground decal, expanding ring, aura mesh 형태 결정 |
| `Audio` | 스킬/BGM/phase 전환 사운드 후보 | 스킬별 SFX 및 보스 BGM 연결 |

현재 임시 후보 정리 위치:

```txt
C:\SL\Igris_Temp
```

## 6. RootMotion / Loop 적용 구조

`MONSTER_ANIM_BIND_DESC` 에 애니메이션별 override 필드를 추가했다.

- `bOverrideLoop`
- `bLoop`
- `bOverrideRootMotion`
- `bUseRootMotion`

`CBody_Monster::Register_AnimationClips()` 에서 override 가 켜진 경우에만 `CModel::Set_AnimationLoop()` / `Set_AnimationUseRootMotion()` 을 적용한다.

현재 강제 정책:

- `Skill_10_Start`: Loop OFF, RootMotion OFF
- `Skill_10_Loop`: Loop ON, RootMotion OFF
- `Skill_10_End`: Loop OFF, RootMotion OFF
- `Skill_04`, `Skill_13`: 기존 `.bin` RootMotion 메타 유지

## 7. HitBox Notify 예정

현재 AI 기준으로 Notify 배치가 필요한 애니메이션:

- Weapon HitBox ON/OFF: `Base_Attack_01`, `Skill_02`, `Skill_04`, `Skill_05`, `Skill_13`, 필요 시 `Skill_01`
- Projectile Spawn: `Skill_06` — `m_Time = 1.8333` 근방을 우선 후보로 두고, 검기 projectile 생성 Notify 배치.
- Area/VFX/Hit: `Skill_11`, `Skill_10_Start/Loop/End`
  - `Skill_10`: warning circle, crack decal, ring wave, shockwave 를 단계별 Notify + 내부 타이머로 분리한다. Notify 는 충전/예고 시작점, 데미지는 fill 완료 시점.

## 15. R6-B 사양 확정 (2026-05-21)

### 사양

- **클립 재생 보장**: Start 클립(`GS_Skill01_KnightKiller_Start`, 약 0.9s) 은 **끝까지 재생** 한다. 자연 블렌딩으로 End 진입이 매끄러우려면 Start 클립을 중간에 자르지 않는 것이 맞다. 1.5s 타임아웃 안은 폐기.
- **충돌 윈도우**: Start 재생 시작 시점부터 **End 로의 블렌딩이 시작되기 전까지** 가 충돌 감지 윈도우다. 윈도우 안에 적과 충돌 발생 시 즉시 Start→Loop 전이 (Loop 도 클립 길이만큼 재생, R6-C 에서 Sphere 히트박스 부착).
- **미충돌 결말**: 윈도우 안에 충돌이 없으면 Start 가 자연 종료되며 자동으로 End → IDLE 로 전이된다 (R6-A 의 기존 OnNotify 자동 전이 유지).
- **전진거리 보정**: Start 클립이 0.9s 로 짧아 RootMotion 만으로는 전진거리가 부족하다. 이동량을 약간 늘려 Player 가 충분히 앞으로 나가도록 보정한다. 두 가지 선택지:

## 16. R6-B 구현 결과 (2026-05-22 마감)

R6-B 코드 + 자산 작업 모두 마감. 인게임 정상 동작 확인.

### 16.1 적용 내역 (코드)

§15 구현 분해 B-0 ~ B-5 + B-6 자산 작업 모두 반영. 추가로 디버그 과정에서 발견된 잔이슈 해결.

### 16.2 디버그 잔이슈 (해결됨)

#### 이슈 #1 — Loop 입력 락 (bIsLoop=true 였음)
- 첫 충돌 후 Loop 진입 → ACTION_FINISHED 영구 미발화 → `SKILL_F+LOOP` (priority 5) 영구 점유 → 모든 입력 막힘
- 원인: `GS_Skill01_KnightKiller_Loop` 클립의 `.bin` 메타 `bIsLoop=true`
- 해결: Editor 에서 Loop `bIsLoop=false` 로 갱신. ACTION_FINISHED 발화 → R6-A 의 LOOP→END 자동 분기 정상 작동

#### 이슈 #2 — Loop/End mesh 가 콜라이더보다 전방 (R6-A 이슈 #2 재발현)
- Start 의 RootMotion 도달 위치를 t=0 기준으로 저작된 Loop/End 클립 → 콜라이더 위치에 mesh 가 +Z 어긋남
- 해결: Start/Loop/End 모두 `bUseRootMotion=true` 로 통일. Loop/End 의 PreTransform 음수 offset bake 도 위치 스냅 코드도 불필요했음

#### 이슈 #3 — 2회차 Loop 재생 안 됨 (t 초기화 누락)
- 1회차 정상 → 2회차부터 충돌 후 Loop 가 즉시 종료 → 곧장 END 진입처럼 보임
- 원인: `CharacterAnimTable.cpp:53` 의 KnightKiller Loop bind 가 `bRestartOnEnter = false`
- 첫 진입 후 t=clip_duration 에 멈춰 있던 상태가 2회차 진입 시 그대로 남아 즉시 종료
- 해결: `bRestartOnEnter = true` 로 수정. SKILL_E LOOP KnightKiller (line 64) 도 동일 패턴이라 선제 수정 (R7 대비)
- **교훈**: Step 진행형 클립 (Start/Loop/End) 의 LOOP 만 bRestartOnEnter=false 면 *재진입 시점이 이전 종료 상태에 머무름* → 한 번만 정상 재생. **Step 진행형 클립의 LOOP 는 기본 bRestartOnEnter=true 가 안전한 디폴트**

## 17. DASH 캔슬 도입 (2026-05-22)

R6-B 검증 도중 사용자 요청으로 추가. "스킬/공격 도중 대시로 캔슬 가능해야 한다" — 일반 액션게임 표준 패턴.

### 17.1 진단 — Priority 비교 방향

`Engine\Private\StateMachine.cpp:66` 에서 `if (NextPolicy.iPriority < CurrentPolicy.iPriority) return false;`

→ **priority 가 클수록 우위** (Next 가 Current 이상이어야 통과). 기존 DASH priority=2 라서 BASIC_ATTACK(3) / GUARD(4) / SKILL_F/Q/E(5) / WEAPON_SWAP(5) 모두 우위에 있어 dash 가 거절됨.

### 17.2 적용 — DASH/BACK_DASH priority 2 → 6

```cpp
// CharacterAnimTable.cpp:81~82
{ CHARACTER_ACTION::DASH,       CHARACTER_ACTION_STEP::NONE, 6, true, ... },
{ CHARACTER_ACTION::BACK_DASH,  CHARACTER_ACTION_STEP::NONE, 6, true, ... },
```

**효과**:
- BASIC_ATTACK / GUARD / SKILL_F/Q/E / WEAPON_SWAP **모두 dash 로 캔슬 가능**
- 피격 (FLOAT/DOWN_RECOVERY/BREAKFALL priority 7) 도중에는 dash 불가 (캔슬 무적 방지)
- DASH 중 DASH 재입력 → 정책은 통과, `Can_ConsumeDashCharge()` 차지 가드로 막힘 (기존 동작 유지)

### 17.3 GUARD 게이트 우회

DASH priority 올린 후에도 GUARD 중 dash 가 작동 안 함. 원인: `Player.cpp:171~175` 의 Locomotion 게이트 (`Is_AttackLocked() / Is_GuardLocked()`) 가 `Update_LocoMotion` 호출 자체를 막아 dash intent 가 priority 비교 단계까지 도달 못 함.

```cpp
// Player.cpp:171 패치
const _bool bLocomotionAllowed =
    (false == m_pStateMachine->Is_AttackLocked() &&
     false == m_pStateMachine->Is_GuardLocked()) ||
    true == Intent.bDashRequested;

if (true == bLocomotionAllowed)
    m_pStateMachine->Update_LocoMotion(Intent);
```

→ dash intent 가 있으면 게이트 우회. `Update_LocoMotion` 의 첫 분기가 dash 처리이므로 즉시 return. RUN/IDLE 자동 전이 같은 부수 효과 없음.

### 17.4 SKILL_F Collider 안전망 (권장)

DASH 가 SKILL_F+START/LOOP 를 캔슬하면 ACTION_FINISHED 가 호출 안 되어 `m_pSkillCollider` 가 켜진 채 남을 수 있음. `Player.cpp::Handle_ActionTransition` 에 가드 권장:

```cpp
if (CHARACTER_ACTION::SKILL_F == eFromAction
    && CHARACTER_ACTION::SKILL_F != eToAction)
{
    Enable_SkillCollider(false);
}
```

> R7 (Q/E) 진입 시 다른 스킬도 sphere 슬롯 공유하면 가드 일반화 (`SKILL 전반 → 비스킬 전이`) 필요.
