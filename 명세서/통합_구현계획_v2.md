## 1. 핵심 아키텍처 결정사항 (유지)

| 항목 | 결정 |
|------|------|
| Assimp 격리 | Editor 전용. Engine/Client는 Assimp 무의존 |
| RTTR 격리 | Editor 전용. Engine/Client는 RTTR 무의존 |
| ImGui/ImGuizmo 격리 | Editor 전용. Engine은 `Set_WorldMatrix` 같은 최소 훅만 노출 |
| 모델 포맷 | FBX → SLMD v2 바이너리(.bin) → Engine은 .bin만 읽음 |
| 씬 포맷 | JSON(편집/확인) + 바이너리(런타임) |
| 씬 로더 위치 | Client DLL |
| 프로젝트 구조 | Engine DLL ← Client DLL ← GameApp EXE / Editor EXE |
| CRT 통일 | 모든 프로젝트 동적 CRT (/MDd Debug, /MD Release) |

### 추가 결정사항 (본 버전에서 명시)

- **SSOT 원칙**
  - 모델 메타데이터(Loop / UseRootMotion / RootBoneName)는 `.bin`이 단일 출처
  - Action → AnimationName 매핑 테이블은 Client(`CharacterAnimTable`, `MonsterAnimTable`)가 단일 출처
  - AnimController는 Action 의미를 모른다. `_uint64 key` 와 `AnimationName` 만 안다
- **Character/Monster 용어 구분**
  - `Character`: 플레이 가능 계열 (조작 주체는 `CHARACTER_CONTROL_TYPE`으로 별도 분리)
  - `Monster`: 적 계열. 행동 축은 `MONSTER_ACTION + MONSTER_PHASE`

## 4. 이후 단계 (목차 중심)

> 아래 단계들은 각각 진입 시 본격 설계 논의를 거친다. 여기서는 목적과 대표 구성 요소만 나열한다.

### 4.1 단계 3 — Player 게임플레이

**목적:** 입력 → 의도 → Action → AnimController로 이어지는 플레이어블 상태머신 완성

**주요 구성**
- 입력 매핑 (이동 / 공격 / 스킬 / 가드 / 패리 / 대시)
- 입력 해석 계층 (원시 키 입력을 즉시 AnimationName/index로 연결하지 않고, gameplay 의미 단위로 해석)
- Player 상태머신 (현재 Action, 다음 Action 큐, Priority 가드)
- 콤보 시스템 (BASIC_ATTACK_01 → 02 → 03 체이닝)
- 무기 장착/해제, `CHARACTER_WEAPON_STATE` 전환
- AnimNotify / 이벤트 (키프레임 타이밍 → gameplay 콜백; 히트박스 on/off, 사운드, 이펙트 트리거)
- 이동 로직: In-place(Walk/Run) 에서 Transform 직접 이동, Root Motion(Dash 등)에서는 Transform 갱신 흡수
- Camera 상호작용 (타겟팅, 카메라 락온 여부)

**실제 키 매핑 초안 (2026-04-21 기준)**
- 이동: `W / A / S / D`
- 시점/바라보는 방향 조작: `Mouse Move`
- 회피: `SPACE` (2026-04-26 정책 갱신 — 이동 입력 유무 기반 분기. 단계3 §4.4.1 참조)
  - `WASD + SPACE` (이동 입력 있음) → `DASH` — 카메라-상대 이동 방향으로 즉시 회전 후 진입
  - `SPACE` 단독 (이동 입력 없음) → `BACK_DASH` — 현재 facing 유지
  - **`S + SPACE` 도 `DASH`** (S 단독도 이동 입력 → 회전 후 후방 DASH). 더 이상 키-조합 기반 BACK_DASH 아님
  - Dash/BackDash 이후에는 짧은 후속 이동 구간을 거쳐 `WALK -> RUN`으로 이어지는 정책 검토
- 일반 공격 / 콤보: `MOUSE_LB`
  - 일정 시간 안에 연속 입력 시 `BASIC_ATTACK_01 -> 02 -> 03`
- 가드: `MOUSE_RB`
  - Guard 성공 시 Parrying 판정
- 일반 스킬: `Q / E`
- 궁극기 스킬: `R`
- 무기 고유 스킬: `F`
- 오버 드라이브: `V`
- 군주화/싱크로 체인: `T`
- QTE 스킬: `LEFT SHIFT`
- 상호작용: `G`
- 물약: `Z`
- 락온: `Mouse Wheel Click`
- 락온 대상 변경: `Mouse Wheel Scroll`
- 동료 헌터 태그 / 그림자 소환: `1 / 2 / 3`

**카메라 / 플레이어 조작 해석** (2026-04-24 개정: 비락온 이동을 **부드러운 회전 이동형(Style C)** 으로 확정)
- 카메라는 플레이어 등 뒤 상단의 추적 카메라를 기본으로 하며, 플레이어는 화면 중앙에 가깝게 유지된다
- `W/A/S/D`는 카메라를 움직이는 입력이 아니라, **플레이어를 조작하는 이동 입력**이다
- `Mouse Move`는 자유 카메라 비행 조작이 아니라, **카메라 Yaw 를 직접 돌리는 입력**이다. Player facing 은 Mouse 에 직결하지 않고 이동 입력 기반으로 간접 결정된다
- **비락온 이동 정책 (Style C — 부드러운 회전 이동)**:
  - WASD 입력은 **카메라 Yaw 기준 월드 방향벡터** 로 해석된다
    - `W` : 카메라 정면(월드) 방향
    - `S` : 카메라 후면(월드) 방향
    - `A` : 카메라 기준 왼쪽(월드) 방향
    - `D` : 카메라 기준 오른쪽(월드) 방향
    - 조합(W+A 등)은 두 방향 벡터의 합 정규화
  - Player 는 결정된 목표 방향으로 **회전 속도(rad/s) 한계 내에서 부드럽게 보간 회전**한다 (`CPlayer::m_fRotationPerSec` 재사용)
  - 회전 진행 중에도 전진 이동은 동시에 진행 (멈춰 돌지 않음). 방향이 반대에 가까울수록 원호 궤적
  - **스트레이프 없음** (비락온 한정). A/D 는 좌우 측면 이동이 아니라 "왼쪽/오른쪽으로 달리며 회전"
- 락온 상태에서는 타겟 기준 회전/스트레이프 정책을 별도로 둔다 (단계 3 비범위)
- `Mouse Y` 입력은 필요 시 카메라 Pitch 제한 조정 축으로 사용하되, 핵심 조작은 `Mouse X` 기반 카메라 Yaw 회전이다

**입력 해석 원칙**
- 키 입력을 곧바로 `AnimationIndex`로 연결하지 않는다
- `Input -> Intent -> Action -> AnimController` 흐름으로 해석한다
- 예:
  - `Mouse Move` → `Intent::LookDelta` (소비자: 카메라 Yaw. Player Yaw 는 직접 받지 않음)
  - `W/A/S/D` → `Intent::MoveVector` (카메라 Yaw 기준 월드 방향. 소비자: Player 가 회전 속도 한계 내 보간 추종 + 전진)
  - `WASD + SPACE` → `Intent::DashForward` → `CHARACTER_ACTION::DASH` (StateMachine 진입 직전 `Player::Face_DirectionImmediately(vMoveDirWorld)` 로 즉시 회전)
  - `SPACE` 단독 → `Intent::DashBackward` → `CHARACTER_ACTION::BACK_DASH`
  - `MOUSE_LB` → `Intent::BasicAttackChain` → 현재 콤보 단계에 따라 `BASIC_ATTACK_01/02/03`
  - `MOUSE_RB` → `Intent::GuardHold` → `GUARD_START / LOOP / END`

**무기 스왑 (`C`) 해석**
- `C`는 단순 무기 토글이 아니라, **항상 들고 있는 두 무기 슬롯 사이의 활성 무기 스왑 입력**으로 본다
- 기본 해석:
  - 현재 활성 무기 `A`
  - `C` 입력
  - 목표 무기 `B`로 전환 요청
  - 전환 액션 재생
  - **`B` 무기 기준 CoreAttack이 즉시 발동**
  - 전환 완료 후 활성 무기 상태가 `B`로 확정
- 따라서 `C`는 일반 공격/스킬처럼 "현재 무기 기준 액션"이 아니라, **목표 무기 기준 액션**이다
- 구현상 `CurrentWeaponState` 와 `PendingWeaponState`(또는 이에 준하는 개념)를 구분하는 방향을 권장한다
- HUD 스킬 UI 변경은 이 전환 결과를 표현하는 후속 UI 단계의 책임으로 둔다 (단계 6에서 구체화)

**상태/표현 분리 원칙**
- 무기 논리 상태 (`CurrentWeaponState`, `PendingWeaponState`)
- 애니메이션 상태 (`CurrentAction`, combo queue, auto return)
- 비주얼 표현 상태 (무기 메쉬 토글, HUD 스킬 UI)
- 위 세 가지를 동일 시점에 강결합하지 않고, gameplay 논리 상태를 먼저 확정한 뒤 비주얼/UI가 이를 표현하는 구조를 유지

**현 단계 주의**
- HUD/Font 시스템은 본 단계와 병행하지 않고 기존 계획대로 후속 단계에서 구현
- 단계 3의 목표는 먼저 "입력/상태/액션/애니메이션" 루프를 안정화하는 것이다

**선행 조건**
- 단계 2-4 (Editor 프리뷰 분리) 완료 권장
- 단계 2-3 (블렌딩)는 품질 향상 축으로 병행 가능

---

### 4.2 단계 4 — NavMesh (Navigation / Cell)

**목적:** Editor에서 맵 위에 이동 가능 영역을 수동 배치하고, 런타임에 이동 제약으로 사용

**주요 구성**
- Engine: `CNavigation`(CComponent), `CCell`(CBase), 필요 시 `CVIBuffer_Cell`
- Editor: NavMesh 편집 모드 (맵 표면 피킹 → 정점 수집 → 3개 모이면 삼각형)
- 자동 Winding 보정 (Cross Product 법선 Y 부호 통일)
- 인접 관리 / 정점 스냅 / 삭제 / 이동
- 저장: 별도 NavMesh 데이터 파일
- 맵 피킹 성능: 규모에 따라 Octree/BVH 검토

**수업 코드 참고**: `9개월차/2~4일차`(`Navigation.h`, `Cell.h`, `VIBuffer_Cell.h`), 5~6일차 AStar 폴더(경로 탐색)

---

### 4.3 단계 5 — 충돌 / 바운딩 / 디버그 드로우

**목적:** 히트 판정 기반 전투 성립을 위한 최소 충돌 시스템

**주요 구성**
- Engine: `CBounding`(abstract), `CBounding_AABB/OBB/Sphere`, `CCollider`(CComponent)
- Engine: `DebugDraw` (DirectXTK `PrimitiveBatch`, `VertexPositionColor`)
- 공격 히트박스: 콤보 단계별 on/off 타이밍 (AnimNotify와 연결)
- 피격 판정: 캐릭터 바디 콜라이더
- 맵/오브젝트 정적 충돌 (선택)
- Editor Viewport 디버그 모드 토글 (콜라이더 와이어프레임 가시화)

**수업 코드 참고**: `9개월차/5~6일차`(`Collider.h`, `Bounding*.h`, `DebugDraw.h`)

---

### 4.4 단계 6 — Font + 2D UI 에디터

**목적:** 텍스트 렌더링과 2D UI 오브젝트 배치/편집

**주요 구성 — Font 시스템**
- Engine: `CFont_Manager`(싱글톤 래퍼, `CGameInstance` 소유), `CCustomFont` (DirectXTK `SpriteBatch/SpriteFont`)
- `Draw(tag, text, pos, color, rot, origin, scale)` 공용 API
- 폰트 에셋 경로 규약

**주요 구성 — 2D UI 에디터**
- Editor: `CPanel_2DCanvas` (게임 해상도 기준 직교 좌표계)
- 기존 `CUIObject` / `UIOBJECT_DESC` 기반으로 배치/이동/리사이즈
- UI 요소: 이미지, 텍스트, 버튼/슬롯 (필요 시 확장)
- UI Scene 파일 별도 관리 (3D Scene과 분리)
- 씬 구조: Base UI(HUD) 자동 로드 + 이벤트성 오버레이 UI(인벤토리, 보스 HP 등)

**수업 코드 참고**: `9개월차/1일차`(`Font_Manager.h`, `CustomFont.h`)

## 6. 규약 요약

### 6.1 SSOT (Single Source of Truth)

| 데이터 | 단일 출처 |
|---|---|
| 애니메이션 Loop / UseRootMotion | `.bin` (퍼-애니메이션 플래그) |
| RootBoneName | `.bin` |
| Action → AnimationName 매핑 | Client 테이블 (`CharacterAnimTable`, `MonsterAnimTable`) |
| Transform (World Matrix) | `CTransform::m_WorldMatrix` (Inspector/ImGuizmo는 폴링) |
| 모델 메쉬/머티리얼 데이터 | `.bin` |

### 6.2 책임 분리

- `CAnimController`: 클립 재생 + 완료 통지 (Action 의미 몰라야 함)
- `CBody_Player` (현재): Table 바인딩 + Policy 기반 자동 복귀
- `CPlayer` 또는 Player 컨트롤러 (단계 3): 입력 → Action 결정, 콤보/Priority 가드
- `AnimNotify` (단계 3): 키프레임 이벤트 → gameplay 콜백 (히트박스, 사운드, 이펙트)

### 6.3 명명/코드 규약 (기존 유지)

- 접두사: `C`(클래스), `m_`(멤버), `ENGINE_DLL`/`CLIENT_DLL`
- 캐스팅 매크로: `ETOUI`, `ETOUI64`
- 해제: `Safe_Delete`, `Safe_Release`, `Safe_AddRef`
- 정수 리터럴 enum 에는 `: _uint` 등 명시
