## 3. 구조 리팩토링 — 설계 의도 합의

### 3.1 문제 인식

기존 `CHARACTER_ACTION` 한 enum 으로 모든 동작 + 모드 + 진행도를 평탄화 → 키맵이 늘어날수록 폭발:
- 콤보 (BASIC_ATTACK_01/02/03) 가 enum 평탄화 → 콤보 단계마다 enum 추가
- GUARD_START/LOOP/END 가 enum 평탄화 → phase 별 enum
- 무기 베리언트 × 액션 = 데카르트곱 매핑 (`.bin` 의 `GS_*` 무기 30종 × 스킬 N개)
- `WEAPON_STATE` 안에 `COMMON` (wildcard) + `UNARMED` (실제 무기 타입) + `FIGHTER` (정체불명) 혼재

### 3.2 솔로 레벨링 키맵 분석으로 도출된 분류

| 키 | 동작 | 분류 |
|---|---|---|
| WASD / SPACE | 이동/회피 | LOCOMOTION |
| MLB | 일반 공격 (콤보) | COMBAT (Action 1개 + 내부 progression) |
| C | 강공격/무기교체 | COMBAT (CORE_ATTACK) |
| Q/E | 일반 스킬 (A/B 슬롯) | COMBAT (NORMAL_SKILL — 슬롯 내부 진행) |
| F | 무기 스킬 | COMBAT (WEAPON_SKILL — 보류) |
| LSHIFT | QTE 스킬 | COMBAT (QTE_SKILL — 보류) |
| R | 궁극기 | COMBAT (ULTIMATE) |
| T | 군주화/싱크로 체인 | COMBAT (LINK_ATTACK / CHAIN_SMASH 후속) |
| V | 오버드라이브 | 스킬 일종 (모드 토글 아님) |
| MRB | 막기 | GUARD |
| Z / G / 1·2·3 / MWheel | 포션/상호작용/태그/락온 | ACTION_LOCKED 또는 시스템 외부 |

### 3.3 3축 분리 결정

기존 1축 (CHARACTER_ACTION 평탄화) → 3축 분리:

```
┌─────────────────────────────────────────────────────────────────┐
│ State  = 전이 규칙 + 입력 허용 범위 결정                           │
│         (StateMachine 의 큰 컨텍스트)                             │
├─────────────────────────────────────────────────────────────────┤
│ Action = 애니메이션 선택 + 한 번성 이벤트 결정                     │
│         (Player 입력 → State 가 허용한 Action 트리거)              │
├─────────────────────────────────────────────────────────────────┤
│ WEAPON_TYPE = 무기 카테고리 (애니 클립 베리언트 선택용)             │
│              (Action 같지만 무기에 따라 클립 다른 경우)            │
└─────────────────────────────────────────────────────────────────┘
```

**State 의 역할은 "입력 차단" 이 아니라 "같은 입력의 의미 차별화"**
예: LOCOMOTION 에서 MLB → COMBAT 으로 전이 + BASIC_ATTACK_01 시작
COMBAT 에서 MLB → 콤보 진행 (BASIC_ATTACK_02)
HIT_STAGGER 에서 MLB → 게이트 차단 (전이 없음)

### 4.1 `Client_Enum.h`

```cpp
enum class CHARACTER_STATE
{
    LOCOMOTION,
    COMBAT,
    GUARD,
    HIT_STAGGER,    // 약 피격 — 곧 회복 (Damage_A/B/C/Breakfall 포함)
    AIRBORNE,       // 공중 (Float_*) — 공중 콤보 받을 수 있음
    DOWN,           // 지면 다운 (Knockdown) — 추가타/회복 가능
    STUN,           // 기절 (지속 상태이상)
    ACTION_LOCKED,  // 단발 잠금 (포션/상호작용/무기교체/태그/UnDraw 등)
    DEAD,
    END
};

enum class WEAPON_TYPE
{
    DAGGER, SWORD, BOW, PISTOL,
    POLE_WEAPON, MAGICAL_WEAPON, TWO_HANDED_WEAPON,
    END  // END = wildcard (무기 무관 클립용)
};

enum class CHARACTER_ACTION
{
    // === LOCOMOTION 한정 (오늘 1단계) ===
    IDLE,
    WALK,             // 컷신/특수 상황 보존 (게임 플레이엔 미사용)
    RUN,
    RUN_FAST,         // DASH 후속 빠른 RUN
    DASH,
    BACK_DASH,
    RUN_END_LEFT,     // 멈출 때 왼발 정지 모션 (FX_Point_L_Foot 가 뒤)
    RUN_END_RIGHT,    // 오른발 정지
    END
};

enum class CHARACTER_TYPE
{
    SUNGJINWOO_ERANK,
    SUNGJINWOO_OVERDRIVE,
    END
};
```

```cpp
_uint64 CBody_Player::Resolve_ActionKey(CHARACTER_ACTION eAction) const
```

## 8. Player_StateMachine LOCOMOTION 새 설계 (코드블록 제시 — 적용 대기)

### 8.1 새 LOCOMOTION 흐름

```
[IDLE]
  └ WASD ────────────────────► [RUN] ────────────► [RUN] (자기보존)
                                  └ WASD 떼기 ────► 발 본 위치 비교
                                                     ├ L 더 뒤 → [RUN_END_LEFT]
                                                     └ R 더 뒤 → [RUN_END_RIGHT]
                                                        └ 종료 ─► [IDLE]
[RUN_END_*]
  └ WASD 재입력 ──────────────► [RUN] (가로채기)

[IDLE/RUN/RUN_FAST]
  └ SPACE + WASD ─────────────► [DASH] (즉시 회전)
                                  └ 종료 ─► WASD 유지 시 [RUN_FAST]
                                            없으면 발 본 비교 → [RUN_END_*]
  └ SPACE 단독 ────────────────► [BACK_DASH]
                                  └ 종료 ─► 동일

[RUN_FAST]
  └ WASD 떼기 ────────────────► RUN 과 동일 (RUN_END_* 분기)
```

### 8.2 발 본 비교 정책

`FX_Point_L_Foot` (idx=27) / `FX_Point_R_Foot` (idx=41) 의 **모델 로컬 z 위치** 비교.

```cpp
CHARACTER_ACTION CBody_Player::Pick_RunEndAction() const
{
    if (nullptr == m_pModelCom)
        return CHARACTER_ACTION::RUN_END_LEFT;

    const _float4x4* pLFoot = m_pModelCom->Get_BoneMatrixPtr("FX_Point_L_Foot");
    const _float4x4* pRFoot = m_pModelCom->Get_BoneMatrixPtr("FX_Point_R_Foot");

    if (nullptr == pLFoot || nullptr == pRFoot)
        return CHARACTER_ACTION::RUN_END_LEFT;

    // _43 = translation.z. 더 작은 z (= 더 뒤) 쪽 발로 멈춤
    return (pLFoot->_43 < pRFoot->_43)
        ? CHARACTER_ACTION::RUN_END_LEFT
        : CHARACTER_ACTION::RUN_END_RIGHT;
}
```

> **검증 필요**: OverDrive 모델 forward 축이 +Z 가정. -Z 면 부등호 반전.

### 9.3 다음 State (COMBAT) 진입 시 결정 거리

LOCOMOTION 검증 끝나면 COMBAT 단계로:

- BASIC_ATTACK 평탄화 유지 vs 단일 enum + 콤보 인덱스 (다시 결정)
- 공격 시 LOCOMOTION → COMBAT 전이 트리거 + 자동 이탈 시점 (UnDraw 자동 처리, IDLE 주석 의도)
- NORMAL_SKILL 슬롯 내부 진행 (Q=A 루트, E=B 루트, CoreAttack 으로 슬롯 전환)
- 무기 베리언트 클립 매핑 정책 (WEAPON_TYPE 별 ANIM_BIND_DESC 다중 등록)
- LINK_ATTACK / CHAIN_SMASH 트리거 정의 (T 키 + 후속 조건)

### 9.4 이후 State 작업 순서 (예상)

1. **LOCOMOTION 검증 마무리** (내일)
2. **COMBAT** — BASIC_ATTACK 콤보 + CORE_ATTACK + NORMAL_SKILL 슬롯
3. **GUARD** — Start/Loop/End 내부 phase + 가드 캔슬 정책
4. **HIT_STAGGER / AIRBORNE / DOWN / STUN** — 피격 분기 (HIT_TYPE 패키지 도입)
5. **ACTION_LOCKED** — POTION / INTERACT / WEAPON_SWAP / TAG / UNDRAW
6. **DEAD** — 종착점

## 11. 학습 / 메모

### 11.1 AssetStudio Dynamic 본 제외 추출

Solo Leveling 캐릭터 모델은 Unity 의 dynamic bone 시뮬레이션을 위해 의상/장식 본을 **별도 GameObject 트리** 로 운영. AssetStudio 가 그대로 평탄 export 하면 dual hierarchy 발생.

**해결**: AssetStudio 추출 시 **Dynamic 계열 제외 옵션** 사용. 다른 헌터 모델 export 시 동일 패턴 적용.

### 11.2 State / Action / Weapon 3축 분리의 본질

State 가 입력 차단이 아니라 "**같은 입력의 의미 차별화 컨텍스트**" 라는 점이 핵심. MLB 한 키가:
- LOCOMOTION 에선 → COMBAT 전이 + BASIC_ATTACK_01 시작
- COMBAT 에선 → 콤보 진행 (BASIC_ATTACK_02)
- HIT_STAGGER 에선 → 차단

이 차별화를 if/else 체인 없이 데이터 (전이 매트릭스 + State 별 입력 핸들러) 로 표현하는 게 목표.

### 11.3 Wildcard fallback 정책의 가치

`WEAPON_TYPE::END = wildcard` 로 두고 `Resolve_ActionKey` 에서 (현재 무기) → (wildcard) 순 fallback 검색. 무기 무관 클립 (LOCOMOTION) 을 무기 종류마다 중복 등록 안 해도 됨. COMBAT 단계에서 무기 베리언트 클립 추가 시 자연스럽게 우선 매칭.

### 11.4 SLMD 분석 도구의 가치

`명세서/_tools/parse_slmd.py` 등 도구가 모델 검증·디버깅 시간을 크게 줄임. 향후 다른 모델 도입 시:
1. `parse_slmd.py` 로 본/메쉬/애니 일괄 덤프
2. `find_bones.py` 로 발/손/무기 점 본 존재 검증
3. `verify_loop_rm.py` 로 Loop/RM 권장값 회귀 확인
