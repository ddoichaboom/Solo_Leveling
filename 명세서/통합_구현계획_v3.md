## 3. 트랙 1 — 추적 카메라 마감 + Style C 이동 정책

### 3.1 개념: 추적 카메라란?

캐릭터를 따라가며 화면 중앙 근처에 유지하는 카메라. **자유 카메라** (Camera_Free, 현재 Editor용) 와 달리 입력이 카메라가 아니라 **캐릭터** 를 움직이고, 카메라는 결과를 따라간다.

핵심 변수 4개:
- **Target**: 따라갈 대상 (Player Transform)
- **Offset**: 타겟 기준 카메라 위치 (등 뒤 + 상단)
- **LookAt**: 카메라가 보는 지점 (보통 타겟 머리 부근)
- **Yaw / Pitch**: 마우스로 회전하는 카메라 자체의 자세

### 3.2 비교 대안

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Direct Follow** | `CamPos = TargetPos + Offset(R(yaw))` 직접 계산 | 단순, 즉각 반응 | 끊김 — 빠른 회전 시 어지러움 |
| **Spring Arm (Lerp)** | 목표 위치를 매 프레임 lerp 추종 | 부드러움, 충돌 회피 가능 | 튜닝 필요 (Stiffness/Damping) |
| **Boom (Unreal)** | 타겟에서 뻗은 막대 끝 카메라 + 충돌 시 줄어듦 | 벽 뒤로 카메라 안 들어감 | 충돌 트레이스 비용. 본 프로젝트는 트랙 4 후에야 가능 |
| **Cinemachine 식 (Unity)** | 데드존 + 락온 + 우선순위 | 영화적 표현 | 본 프로젝트 규모엔 과함 |

**본 프로젝트**: **Spring Arm 즉시 추적 + 충돌 복귀용 Distance Lerp** 로 시작. 평상시에는 타겟 기준점과 카메라 거리를 즉시 유지해 액션 조작감을 안정화하고, 트랙 4 이후 벽 충돌로 arm 길이가 줄어든 경우에만 원래 거리 복귀를 Lerp 처리한다. Target/Pivot smoothing 은 이동 중 고무줄 느낌을 만들 수 있어 트랙 1 범위에서 제외한다.

### 3.3 Style C 이동 정책 (계승, v2 §4.1 결정)

이미 결정된 사항:
- WASD 는 **카메라 Yaw 기준 월드 방향벡터** 로 해석
- Player 는 목표 방향으로 **회전 속도 한계 내 보간 회전** + 동시 전진
- 비락온 한정 **스트레이프 없음** (A/D 는 좌우 측면이 아니라 "왼쪽/오른쪽으로 달리며 회전")
- Mouse X → 카메라 Yaw (Player 아님)

현재 코드 기준 필요한 것: **기존 Camera_Follow/SpringArm 마감 + Intent 구조 정리 + Player 삭제/재생성 시 target 재바인딩**.

### 3.4 본 프로젝트 적용

```
[입력]  Mouse X Δ ──→ Camera_Follow.Yaw 직접 갱신
        WASD       ──→ IntentResolver: Camera Yaw 기준 월드 방향 산출
                       → PLAYER_INTENT_FRAME::vMoveDirWorld (정규화)

[Player] Apply_MoveIntent(vMoveDirWorld):
         목표 Yaw = atan2(vMoveDirWorld.x, vMoveDirWorld.z)
         현재 Yaw 와 차이를 m_fRotationPerSec 한계 내 보간
         회전 후 Position += vMoveDirWorld * fSpeed * dt  (단순 직진 합성)

[Camera_Follow] Update:
         TargetWorldPos = Layer_Player 의 Player Transform 자동 검색/재바인딩
         TargetPoint = TargetWorldPos + HeightOffset
         LookDir = Yaw/Pitch 로 계산
         Eye = TargetPoint - LookDir * CurrentDistance
         CurrentDistance 는 평상시 IdealDistance. 트랙 4 충돌 시 줄어든 거리에서 복귀할 때만 Lerp
```

**핵심 결정**:
- `Camera_Free` 프로토타입은 Client Loader 에 유지. Editor 가 Client 런타임 리소스를 호스팅하므로 제거하지 않는다.
- GamePlay 런타임 인스턴스는 `Camera_Follow` 만 생성. `Camera_Free` 인스턴스는 Editor `Bootstrap_EditScene()` 에서만 생성한다.

## 4. 트랙 2 — NavMesh + Scene 저장/로드

### 4.1 개념: NavMesh 란?

**Nav**igation **Mesh**. 캐릭터(보통 AI) 가 이동할 수 있는 영역을 **삼각형 셀들의 집합** 으로 표현. 각 셀은 **인접 정보** (어느 변에 어떤 셀이 있는지) 를 들고 있어 그래프 탐색이 가능하다.

핵심 사용처:
1. **이동 제약**: 셀 밖으로 안 나가게 (벽/낭떠러지)
2. **경로 탐색**: 시작 셀 → 목표 셀까지의 셀 시퀀스 (A*)
3. **AI 추적**: Monster 가 Player 셀까지 경로 계산해 추적

### 4.2 비교 대안

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **자동 생성 (Recast/Detour)** | 메쉬에서 voxel-bake 로 NavMesh 자동 산출 | 손 댈 필요 없음. 산업 표준 (Unity 내장, Unreal NavSystem) | 라이브러리 의존, 본 학습 목적과 거리 멈 |
| **수동 셀 편집 (수업 코드, 본 프로젝트)** | Picking 으로 정점 찍어 셀 직접 만듦 | 학습용으로 좋음, 의존 없음, 제어 완전 | 소규모 맵에만 현실적, 정밀도 사용자 책임 |
| **그리드/노드** | 격자 셀 ON/OFF + 4/8 방향 인접 | 단순 | 자유 형태 표현 어려움, 대각선 비용 부정확 |
| **Voronoi / 시각적 그래프** | 장애물 분포에서 점들 산출 | 좁은 통로 표현 좋음 | 구현 복잡 |

**본 프로젝트**: **수동 셀 편집** + Picking + 정점 스냅 + Winding 자동 보정. 수업 코드 9개월차 2~5일차 패턴 계승. 학습 목적에 가장 부합.

### 4.3 본 프로젝트 적용 — 편집 워크플로우

수업 코드 핵심 클래스:
- `CCell` (`9개월차/4일차/Engine/Public/Cell.h:7`) — 정점 3개, 법선 3개, 인접 인덱스 3개, 평면 1개
- `CNavigation` (`...Navigation.h:7`) — `vector<CCell*>` + 현재 셀 인덱스 + 부모 매트릭스
- `CVIBuffer_Cell` (3일차+) — 셀 가시화용 와이어프레임
- A* (5일차/AStar 폴더) — `Make_Route(iGoalIndex)` → `list<_vector>`

**우리가 추가할 것 (수업 코드와 다른 점)**:
- Editor Picking 기반 정점 입력 (수업 코드는 외부 도구로 만든 데이터 파일 fread)
- 정점 스냅 (월드 거리 임계값 기준 기존 정점 재사용)
- Winding 자동 보정 (Cross.y 부호로 CCW 통일)
- NavMesh 데이터 별도 파일 저장 (`.navdata` 형식 — 또는 Scene I/O 와 통합)

**편집 흐름 (확인된 사양)**:
1. Editor "NavMesh Edit Mode" 토글
2. 맵 메쉬에 raycast → 표면 좌표 채택
3. 임계 거리 안에 기존 정점 있으면 **기존 정점 ID 재사용**, 없으면 **신규 정점 추가**
4. 같은 셀 안에서 동일 정점 ID 가 중복되면 생성 거부
5. 정점 3개가 모이면 삼각형 면적 검사 후 **자동 셀 생성** (Winding 보정 후)
6. Cell 생성/삭제/정점 이동 후 Neighbor 전체 재계산
7. 입력 미완(1·2번째)일 때 미리보기 + ESC 취소
8. Cell 선택 / 삭제 / 단일 정점 드래그 (인접 셀 영향)
9. 데이터 변경 명령에 대한 Undo/Redo
10. 저장 (`.navdata`), 다음 부팅 시 로드

### 4.6 미결 / 학습 포인트

- **NavMesh 와 맵 충돌 분리**: NavMesh 는 "어디까지 갈 수 있나" (큰 영역), 트랙 4 의 충돌은 "이 점이 닿았나" (정밀). 둘은 별개 시스템 — 혼동 금지.
- **Player 도 NavMesh 제약?**: 본 프로젝트에서는 Player 도 NavMesh 위를 이동한다. Track 2 에서 Player 이동을 NavMesh 로 먼저 검증하고, Track 5 Monster/AI 가 같은 구조를 재사용한다.
- **셀 정점 공유 vs 비공유**: 공유 시 인접 그래프 자동, 한 정점 수정이 여러 셀 영향. 비공유 시 인접은 좌표 비교 필수, 수정 영향 격리. 수업 코드는 공유 — **본 프로젝트도 공유** 권장.
- **A* 휴리스틱**: 셀 중심 간 직선거리 (Manhattan/Euclidean). 본 프로젝트는 Euclidean 충분.
- **Scene I/O 포맷 versioning**: 처음부터 magic + version 박을 것. 누락 필드는 빈 값 폴백 (SLMD 패턴). 학습 포인트.
- **(학습)** Cross product 의 Y 부호로 winding 판정하는 법: `(B-A) × (C-A)`, .y > 0 이면 CCW (위에서 봤을 때).

### 4.7 Decision Log

#### 2026-05-06 — NavMesh 소유권 / Agent 컴포넌트 구조
- **결정**: NavMesh 원본 데이터는 Scene/Level 쪽에서 하나만 소유한다. Player, Monster 처럼 길찾기 또는 이동 제약이 필요한 객체는 `NavigationAgent` 컴포넌트를 소유한다.
- **구조**: `Player/Monster -> CNavigationAgent` 는 기존 컴포넌트 패턴처럼 `Add_Component` / `Safe_Release` 대상이다. `CNavigationAgent -> CNavMesh` 는 NavMesh 를 생성/삭제하지 않지만, 멤버 포인터로 장기 보관하므로 `Safe_AddRef` / `Safe_Release` 로 참조 수명을 보장한다.
- **이유**: NavMesh 는 캐릭터의 PartObject 가 아니라 맵 이동 데이터다. 데이터는 하나만 두고, 객체별 상태만 Agent 에 분리해야 Editor 저작, Player 검증, Monster 확장이 모두 자연스럽다.
- **초기 바인딩**: 현재 `CGameInstance::Add_GameObject()` 는 생성 객체 포인터를 반환하지 않으므로, 1차 구현은 Agent 가 `Layer_NavMesh` 또는 Scene NavMesh 서비스를 조회해 런타임 바인딩하는 방식으로 시작한다.

#### 2026-05-06 — Player NavMesh 적용 범위
- **결정**: Track 2 에서 Player 도 NavMesh 위를 이동하도록 한다.
- **이유**: Player 이동을 먼저 NavMesh 로 제한하면 Editor 저작 결과를 즉시 검증할 수 있고, Track 5 Monster/AI 가 같은 NavigationAgent 구조를 재사용할 수 있다.
- **1차 이동 정책**: 이동 후보 위치가 현재 Cell 또는 Neighbor Cell 안이면 이동 허용 + 높이 보정. 막힌 Edge 이면 이동 차단. Sliding 은 Track 2 후반 또는 충돌 Track 이후 재검토한다.

#### 2026-05-06 — Cell 저작 데이터 구조
- **결정**: Editor NavMesh 는 공유 정점 풀 기반으로 관리한다. Cell 은 좌표 3개를 직접 들기보다 `VertexIndex[3]` 를 들고, Neighbor 는 `NeighborIndex[3]` 로 관리한다.
- **정점 스냅**: Picking 좌표가 기존 정점의 Snap Radius 안에 있으면 좌표를 복사하는 것이 아니라 기존 정점 ID 를 재사용한다.
- **Winding 보정**: 3정점이 모이면 `(B-A) x (C-A)` 의 Y 부호로 방향을 판정하고, 반대 방향이면 두 정점 인덱스를 swap 해 Cell winding 을 통일한다.
- **생성 방어 조건**: 동일 Cell 내 중복 정점 ID 금지, 면적이 너무 작은 삼각형 금지, Cell 생성/삭제/정점 이동 후 Neighbor 전체 재계산.
- **이유**: 공유 정점 기반이어야 정점 드래그 시 인접 Cell 이 함께 변형되고, Neighbor 계산도 "같은 정점 ID 2개 공유" 기준으로 안정화된다.

#### 2026-05-06 — Cell 선택 / 삭제 / Undo
- **결정**: Track 2 Editor 범위에 Cell 선택, Cell 삭제, Undo/Redo 를 필수 기능으로 포함한다.
- **Undo 대상**: Cell 생성, Cell 삭제, 정점 이동, NavMesh Clear 같은 데이터 변경 작업.
- **Undo 비대상**: Hover, Cell 선택, 생성 중인 1~2개 Preview 정점은 1차 구현에서 Undo 대상에서 제외한다.
- **초기 구현**: Undo/Redo 는 Command delta 방식이 아니라 NavMesh 전체 Snapshot 방식으로 시작한다. NavMesh 규모가 작고, 삭제/정점 이동/Neighbor 재계산의 복원 안정성이 더 중요하기 때문이다.
- **후속 액션**: NavMesh 규모가 커지거나 편집 명령이 복잡해지면 Snapshot 방식에서 command delta 방식으로 최적화한다.

## 5. 트랙 3 — Font + 2D UI 에디터

### 5.1 개념

게임 화면에 **텍스트** (HP 숫자, 메뉴, 데미지 폰트) 와 **2D 이미지** (UI 패널, 아이콘, 로고) 를 그리는 인프라.

3D 캐릭터는 World 공간에서 그려지지만 UI 는 **Screen 공간** (혹은 Orthographic projection) 에서 그린다. 깊이/조명 영향 없이 픽셀 단위 정확하게 배치.

### 5.2 비교 대안

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **DirectXTK SpriteFont** (수업 코드, 본 프로젝트) | `.spritefont` 파일을 makespritefont 도구로 미리 생성, SpriteBatch 로 그림 | 의존 가벼움, 즉시 동작 | 한글 폰트 별도 생성 필요, 동적 글리프 불가 |
| **FreeType + Atlas** | TTF 파일에서 동적으로 글리프 래스터화 | 어떤 폰트든, 어떤 크기든 | 구현 복잡, 학습 비용 큼 |
| **Bitmap Font** | 미리 그린 PNG + 좌표 테이블 | 단순, 게임 룩에 맞춤 | 크기 변경 어려움 |
| **ImGui** | (이미 Editor 에 있음) | Editor 패널로 충분 | 게임 런타임 UI 에는 부적합 |

**본 프로젝트**: **DirectXTK SpriteFont** 시작. 한글 게임이라 spritefont 도구 사용법 학습 필요. 후속에 FreeType 으로 갈지는 트랙 마감 후 판단.

### 5.5 미결 / 학습 포인트

- **레벨 전환 흐름**: 현재는 `CLevel_GamePlay` 만 있음. Logo → Loading → GamePlay 흐름은 `CLevel_Manager::Open_Level` 로 처리. 비동기 로드는 `CLoader` 스레드 (이미 있음). `CoInitializeEx(nullptr, 0)` 필수 (CLAUDE.md Gotchas).

- **Logo/Loading 이미지 에셋**: 원본 게임 Unity 번들에서 추출 진행 중. Texture2D (SpriteAtlas) PNG 활용 예정.
- **CPanel_2DCanvas 방식**: 3D Viewport 와 독립된 별도 ImGui 윈도우. 레벨 전환 불필요.
- **(학습)** Orthographic projection: World 좌표 (-1, +1) 또는 (0, screen_size) 직접 매핑.

### 5.6 Decision Log

#### 2026-05-11 — CUI_Image / CUI_SpriteAnim 클래스 분리
- **결정**: 정적 이미지(`CUI_Image`)와 아틀라스 스프라이트 애니메이션(`CUI_SpriteAnim`)을 별도 클래스로 분리한다.
- **이유**: 아틀라스 애니메이션은 Texcoord UV Offset/Scale 계산 및 프레임 관리 로직이 필요하여 정적 이미지와 책임이 다름. 한 클래스에 섞으면 정적 이미지 사용 시 불필요한 멤버가 포함됨.
- **구조**:
  - `CUI_Image` — 정적 이미지 렌더. Texcoord 고정.
  - `CUI_SpriteAnim` (`CUI_Image` 상속) — 아틀라스 컬럼/행, 프레임 속도, UV Offset 계산. Shader에 `g_vUVOffset` / `g_vUVScale` pass 추가.
- **Track 3 범위**: `CUI_Image`만 구현. `CUI_SpriteAnim`은 실제 필요 시점(컷씬 or 아틀라스 UI 사용 시)에 추가.

#### 2026-05-11 — 3-H (Editor 2DCanvas) 필수 항목으로 격상
- **결정**: `CPanel_2DCanvas` 를 선택이 아닌 필수 Step 으로 변경.
- **이유**: NavMesh 씬 데이터처럼 UI 배치도 에디터 → SceneData → 런타임 로드 흐름을 일관되게 유지해야 함. 하드코딩 방지.

## 6. 트랙 4 — 충돌 / Bounding / DebugDraw

### 6.1 개념

**Bounding** = 물체를 감싸는 단순 도형 (Sphere/AABB/OBB). 정밀한 메쉬-메쉬 충돌은 비싸므로 **단순 도형끼리 먼저** 검사.

**Collider** = Bounding 을 들고 있는 컴포넌트. 매 프레임 Transform 으로 업데이트.

**DebugDraw** = 충돌 형태/광선/그리드 등을 와이어프레임으로 시각화 (DirectXTK `PrimitiveBatch` + `BasicEffect`).

### 6.2 비교 대안

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **PhysX / Bullet / Jolt** | 산업 표준 물리 엔진 | 물리 시뮬레이션, 안정성 | 본 학습 목적과 거리 멈, 의존 무거움 |
| **DirectXTK BoundingSphere/Box/OrientedBox** (수업 코드, 본 프로젝트) | DirectXMath 기본 도형 + Intersects 함수 | 의존 가벼움, 학습용 | 정밀 충돌 / 물리 응답 없음 |
| **수동 GJK/SAT** | OBB-OBB 등 직접 구현 | 학습 깊이 | 구현 비용 큼 |

**본 프로젝트**: **DirectXTK Bounding** 시작. PrimitiveBatch 로 와이어프레임. 수업 코드 6일차 패턴.

### 6.3 본 프로젝트 적용

수업 코드 6일차 핵심:

  - Capsule (산업 표준) 은 DirectXTK 미지원이라 직접 합성(Sphere×2 + Line) 필요 → 학습 비용 큼. 트랙 4 1차에선 제외.
  - **후속**: 트랙 5 검증 후 OBB 가 부족(머리/발 분리 판정 필요 등)하면 Capsule 합성 도입 검토.
- **결정 2 — 공격 히트박스 시공간 = 신규 Notify `ATTACK_HITBOX_ON / ATTACK_HITBOX_OFF`**:
  - 기존 `COMBO_WINDOW_OPEN/CLOSE` 는 **입력 윈도우** (다음 콤보 입력 받기). 히트 윈도우와 의미 분리 필요.

- **결정 3 — 무기 히트박스 부착 = 무기 본 2개(칼날 시작/끝) 기반 OBB**:

  - 매 프레임 두 본의 `CombinedTransformation × Player World` 로 월드 좌표 산출.
  - 두 점을 잇는 선분 → OBB 로 감싸기 (길이 = 두 점 거리, 두께 = 코드 상수 or `CWeapon` 멤버, 회전 = 칼날 방향).

  - 트레일 메쉬 자체 구현은 트랙 4 범위 외 — 본 좌표 노출 API 까지만 트랙 4 에서 마련.
- **결정 4 — Collision_Manager 시스템 도입**:
  - **위치**: Engine DLL (`Engine/{Public,Private}/Collision_Manager.{h,cpp}`). `CGameInstance` 가 보유 — `Initialize_Engine` 에서 Create, `Release_Engine` 에서 정리. 다른 매니저(`Object_Manager`, `Renderer`, `Font_Manager`)와 동일 패턴.
  - **등록 방식 = 매 프레임 등록 (Renderer 패턴)**:
    - Collider 가 `Late_Update` 에서 `Collision_Manager::Add_Collider(eGroup, this)` 호출 → AddRef.
    - Manager 의 Update 단계에서 페어 검사 → 검사 후 group list `clear()` + Safe_Release.
    - 가시성 = 등록 자체. 별도 활성 플래그 불필요. GameObject 소멸 시 자동 해제 (dangling 위험 0).
    - 영구 등록 + bActive 플래그 안은 거부 — 명시적 Unregister 누락 시 dangling 위험 + 기존 컨벤션과 결이 다름.
  - **결과 노출 = 콜백 전면 (OnHitEnter / OnHitStay / OnHitExit)**:
    - Manager 가 **이전 프레임 페어 set** 캐시 보유 → 현재 프레임 페어와 diff 계산 → 상태 변화 발화.

    - 콜백 미등록 collider 는 silent — 등록 비용 0.
    - 폴링은 일부러 노출 안 함. 필요 시점에 추가 (현재 게임 로직은 OnEnter 1회 = 데미지 패턴이 주류).
  - **그룹 매트릭스**: enum `COLLISION_GROUP { PLAYER_BODY, PLAYER_ATTACK, MONSTER_BODY, MONSTER_ATTACK, MAP_STATIC, END }`. Manager 가 `m_CollisionMatrix[GROUP][GROUP]` (bool) 보유 — 어떤 그룹끼리 검사할지 제어. 초기 매트릭스: PLAYER_ATTACK ↔ MONSTER_BODY, MONSTER_ATTACK ↔ PLAYER_BODY 만 true.

## 7. 트랙 5 — Monster + 잔여 Player

### 7.1 개념

**적 캐릭터** + **AI 상태머신**. Player 의 `CPlayer_StateMachine` 과 같은 패턴이지만 입력 대신 **AI 결정** 이 다음 액션을 정한다.

세 가지 AI 패러다임:
| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **FSM (Finite State Machine)** | 상태 + 전이. 본 프로젝트 Player 와 같은 패턴 | 단순, 디버깅 쉬움 | 상태 폭발, 복잡 행동 표현 어려움 |
| **Behavior Tree** | 트리 노드 (Sequence/Selector/Decorator) | 모듈화, 디자이너 친화 | 학습 비용, 본 프로젝트 규모엔 과함 |
| **Utility AI / GOAP** | 효용 함수로 행동 점수화 | 자연스러운 결정 | 튜닝 까다로움 |

**본 프로젝트**: **FSM** (Player 와 같은 패턴 재사용). 단일 몬스터 6~8 상태면 FSM 으로 충분.

### 7.2 본 프로젝트 적용

기본 상태 (수업 코드 / 일반 패턴):
- **Idle** — 대기, 주변 감지
- **Patrol** — 정해진 경로 순찰 (NavMesh 사용)
- **Chase** — Player 발견 시 추격 (NavMesh A*)
- **Attack** — 사거리 안 들어왔을 때
- **Hit** — 피격 반응
- **Stun** — 경직
- **Death** — 사망

**선행 인프라**:
- 트랙 2 NavMesh — Patrol/Chase 경로
- 트랙 4 Collider — 사거리 판정, 피격 검출

### 7.3 잔여 Player 작업 (트랙 5 후반부)

Player 1차 마감 시 보류한 항목들. 이제 적이 있으니 검증 가능:

### 7.6 Decision Log

#### 2026-05-11 — Monster 클래스 계층 / Body 공통화
- **결정**: `CMonster` 는 `CContainerObject` 를 상속하고, 실제 런타임 생성 대상은 `CNormal_Monster`, `CElite_Monster`, `CBoss_Monster` 로 나눈다.
