## 2. 설계 결정

### 2.1 NavMeshObject 역할

`CNavMeshObject` 는 NavMesh 원본 데이터를 Scene/Layer 에 올리기 위한 GameObject 다. `CMapObject` 에 종속시키지 않고 별도 GameObject 로 둔 이유는 Editor 선택, 저장/로드, 와이어프레임 렌더, Player/Monster Agent 바인딩 단위를 명확히 하기 위해서다.

대안 비교:

| 대안 | 장점 | 단점 | 판단 |
|---|---|---|---|
| `CMapObject` 가 `CNavMesh` 소유 | 맵과 같이 생성되어 단순함 | NavMesh 가 특정 맵 오브젝트에 종속. Editor 저작/교체/저장이 불편 | 제외 |
| `Level_GamePlay` 멤버로 보관 | 접근이 직접적 | Object/Layer 시스템 밖이라 Editor 선택/렌더/저장 단위가 애매함 | 제외 |
| `CGameInstance` 전역 서비스 | 어디서든 조회 쉬움 | Scene 별 NavMesh 수명과 소유권이 숨겨짐 | 후순위 |
| `CNavMeshObject` 별도 GameObject | Layer, Render, Picking, 저장 단위가 명확 | 조회/바인딩 경로가 필요 | 채택 |

### 2.2 RefCnt 규칙 적용

- `NavMeshObject -> CNavMesh`: `Add_Component()` 경로로 멤버 포인터를 받으므로 `Add_Component()` 내부 `Safe_AddRef` 가 멤버 보관분을 처리한다. `CNavMeshObject::Free()` 에서 `Safe_Release(m_pNavMeshCom)` 한다.
- `NavigationAgent -> CNavMesh`: 외부 포인터를 `Bind_NavMesh()` 로 직접 멤버 보관하므로 `Bind_NavMesh()` 안에서 직접 `Safe_AddRef` 하고, 교체/해제 시 `Safe_Release` 한다.
- `CCell::m_pVertices`: `vector<_float3>*` 데이터 포인터이므로 RefCnt 대상이 아니다.

### 2.3 `<cmath>` 정책

Track 2 중에는 `Engine_Defines.h` 의 `<cmath>` 공통 include 를 유지한다. 기존 Engine/Client 코드가 수학 함수 include 를 공통 define 경유로 사용하는 구조이므로, include hygiene 정리는 NavMesh 작업 범위 밖의 별도 정리 Task 로 분리한다.
