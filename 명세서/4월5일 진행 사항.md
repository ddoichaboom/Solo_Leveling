## Trouble Shooting

### 1. 순환 참조 메모리 누수 (Step 5~6 적용 후 발생)

**증상**: 프로그램 종료 시 Panel_Manager, 모든 Panel, GameInstance, ContentBrowser 경로 등 대량 메모리 누수 발생

**원인**: CPanel 생성자에서 `Safe_AddRef(m_pPanel_Manager)` + Panel_Manager가 m_Panels에서 Panel을 소유 → 순환 참조
- Panel_Manager RefCount = EditorApp(1) + 5개 Panel(5) = 6
- `CPanel_Manager::DestroyInstance()`는 Release 1회만 수행 → RefCount 5로 남아 절대 해제 안됨

**해결**: `Release_Panels()` 패턴 도입 (Engine의 `Release_Engine()`과 동일 원리)
- `CPanel_Manager::Release_Panels()` — 선택 상태 해제 + 모든 Panel Release + map clear
- `CEditorApp::Free()`에서 `m_pPanel_Manager->Release_Panels()` 호출 후 `DestroyInstance()`
- Panel에서 Panel_Manager AddRef는 유지 (정상적인 참조 관리)

**원리**: COM 스타일 참조 카운팅에서 소유 계층 간 순환이 생기면 수동으로 끊어줘야 함. GameInstance::Release_Engine()이 정확히 같은 문제를 해결하는 방식.

### 2. CVIBuffer PICK_DATA 복사 생성자 누락 (Step 8 구현 중 발견)

**증상**: (잠재적) Clone 후 원본 또는 Clone이 Free되면 댕글링 포인터 → 크래시

**원인**: CVIBuffer 복사 생성자가 `m_pPickData` 포인터를 얕은 복사 → 원본과 Clone이 같은 PICK_DATA 공유 → 한쪽 Free 시 이중 해제

**해결**: 복사 생성자 본문에서 PICK_DATA 깊은 복사
- 이니셜라이저 리스트가 아닌 생성자 본문에서 new + memcpy
- CTexture Clone의 SRV Safe_AddRef와 같은 원리 (COM=AddRef, 일반 힙=깊은 복사)

### 3. VIBuffer_Rect 인덱스 타입 불일치

**증상**: (잠재적) PICK_DATA 인덱스 값이 깨져서 피킹 시 잘못된 삼각형 참조

**원인**: VIBuffer_Rect 인덱스 = `_ushort`(2byte, R16_UINT), PICK_DATA 인덱스 = `_uint`(4byte). memcpy하면 바이트 크기 불일치로 데이터 깨짐

**해결**: for문으로 개별 `static_cast<_uint>` 변환 복사. VIBuffer_Terrain은 원본이 `_uint`이므로 memcpy 가능.

## 핵심 설계 결정 기록

### 피킹 알고리즘: TriangleTests::Intersects (삼각형 단위)

**선택 이유**: AABB는 회전/복잡 형상에서 빈 공간 오선택. 삼각형은 실제 메시와 정확히 일치.

**CPU 정점 보존 방식 (대안 B)**:
- 대안 A (GPU Map/Unmap): 매 클릭마다 GPU↔CPU 왕복 — 느림
- **대안 B (CPU 사본 유지)**: Initialize 시 Position+Index 별도 복사 — 메모리 추가 사용, 즉시 접근
- 대안 C (별도 Picking 버퍼): 구현 복잡, 현재 단계에서 과도

**PICK_DATA 구조체로 캡슐화**:
- nullptr 체크 하나로 피킹 가능 여부 판단 (별도 bool 불필요)
- Engine 소속이므로 Editor뿐 아니라 인게임 3D 상호작용에도 재사용 가능

**역할 분담**:
- 베이스(CVIBuffer): m_pPickData 소유, Pick() 실행, Free() 해제, 복사 생성자 깊은 복사
- 파생 클래스: Initialize_Prototype()에서 PICK_DATA 생성 + 자기 정점 타입에서 Position 추출
- 정점 타입이 다르므로(VTXTEX/VTXNORTEX/VTXMESH) 베이스에서 일괄 추출 불가

**피킹 공간 변환 전략**: Ray를 로컬로 변환 (Ray 1개 변환 vs 정점 N개 변환 → 효율적)

### 상용 엔진과의 비교

| 방식 | 사용처 | 전제 조건 |
|------|--------|-----------|
| 물리 엔진 Raycast | Unity/Unreal 런타임 | PhysX 등 물리 엔진 |
| Hit Proxy (Color ID) | Unreal Editor | 별도 렌더 패스 |
| **CPU 정점 + 삼각형 교차** | 물리 엔진 없는 커스텀 엔진 | 정점 데이터 보존 |

현재 프로젝트에 물리 엔진이 없으므로, 물리 엔진의 MeshCollider가 하는 일(CPU 삼각형 데이터 보존 → Ray 교차)을 CVIBuffer 레벨에서 직접 수행.

## Step 8 구체화 상세 (개념)

### Screen → World Ray 변환 과정

```
[1] Screen Space (픽셀: 0~Width, 0~Height)
    ↓ 정규화 + Projection 역변환
[2] View Space (카메라 원점 기준 방향)
    ↓ View 역행렬
[3] World Space (Ray Origin + Direction)
```

- Proj._11 = 1/tan(fovX/2), Proj._22 = 1/tan(fovY/2)
- x = (2*mouseX/width - 1) / Proj._11,  y = (-2*mouseY/height + 1) / Proj._22
- XMVector3TransformCoord: 점(w=1) 변환, XMVector3TransformNormal: 방향(w=0) 변환

### World Ray → Local Space 변환

- WorldMatrix 역행렬 적용: Ray를 오브젝트 로컬 공간으로 변환
- Normalize 필수: 스케일이 포함된 역행렬 적용 후 방향 벡터 길이 변함 → 거리 왜곡 방지

### TriangleTests::Intersects

- DirectXCollision.h 포함 함수
- 인덱스 3개씩 묶어 삼각형 순회, 가장 가까운 교차점 선택
- 반환 거리는 로컬 공간 → 월드 교차점으로 변환하여 오브젝트 간 비교

### ImGui Viewport 마우스 좌표

- GetCursorScreenPos(): Image 위젯 직전 호출 → 이미지 화면 좌상단
- IsItemHovered() + IsMouseClicked(Left): 뷰포트 Image 위 좌클릭 감지
- 로컬 좌표 = 절대 마우스 위치 - Image 좌상단
